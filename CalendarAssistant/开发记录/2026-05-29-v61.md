# 2026-05-29 开发记录 (v61)：发布包体积优化与本地模型版取舍

## 一、本次目标

本次开发围绕发布包体积和本地模型版本策略展开。

主要目标：

1. 缩减标准版 APK 体积，降低安装和分发成本。
2. 评估是否继续维护带本地模型能力的 `local-model/main` 分支。
3. 验证 R8 / 资源压缩 / JNI 压缩对启动、数据库、设置、识别等主链路的影响。
4. 保留从本地模型版覆盖安装到标准版后的模型残留清理能力。
5. 记录本地模型版尝试、失败原因和最终取舍，便于后续排查。

当前主线状态：

- `main`：标准版主线，后续唯一维护和发布分支。
- `local-model/main`：本地模型版历史分支，暂时保留作为排查参考，不再作为发布分支维护。

---

## 二、标准版移除本地模型能力

### 2.1 背景

早期版本曾接入 LiteRT-LM 本地模型能力，标准版 APK 体积一度接近 90 MiB。虽然仓库不内置模型权重，但 LiteRT-LM 运行库、`ai-engine` 模块、ML Kit OCR、未压缩 Dex 等仍显著增加包体。

为了降低标准版体积和维护成本，决定先在 `main` 分支移除本地模型能力。

### 2.2 移除范围

标准版移除内容包括：

- 从 `settings.gradle.kts` 移除 `:ai-engine` 模块。
- 从 `app/build.gradle.kts` 移除 `implementation(project(":ai-engine"))`。
- 移除本地模型管理和导入入口。
- 移除 LiteRT-LM 客户端和本地模型 Provider。
- 移除本地模型前台服务、日志调度、测试入口等标准版不再使用的链路。
- `RecognitionProviderFactory` 标准版固定走远程/API 识别路径。
- `AiModelConfig` 标准版只校验远程模型配置。

保留能力：

- 云端文本识别。
- 云端多模态识别。
- ML Kit OCR。
- 剪贴板码类识别。
- 无障碍截图识别。
- 天气、小组件、附件、备份等标准功能。

相关提交：


## 三、Release 包体积优化

### 3.1 体积来源分析

旧 release 包体积较大的主要原因：

- Release 未开启 R8，Dex 体积接近 56 MiB。
- Native libs 未压缩打包。
- LiteRT-LM native 运行库较大。
- ML Kit OCR 自带 native 库和 assets。
- `material-icons-extended`、Ktor、Compose 等依赖在未压缩时占用较多 Dex。

标准版移除本地模型后，APK 约为：

```text
68.95 MiB
```

### 3.2 开启 R8 与资源压缩

在 `app/build.gradle.kts` 的 release 配置中开启：

```kotlin
isMinifyEnabled = true
isShrinkResources = true
```

开启后标准版 APK 降至约：

```text
20.79 MiB
```

### 3.3 JNI 压缩打包

继续开启 JNI 压缩：

```kotlin
packaging {
    jniLibs {
        useLegacyPackaging = true
    }
}
```

最终标准版 unsigned release APK 约为：

```text
14.42 ~ 14.43 MiB
```

主要组成：

- Dex：约 6.5 MiB。
- Native libs：压缩后约 4.2 MiB。
- assets：约 1.9 MiB。
- resources.arsc：约 1.2 MiB。

### 3.4 ProGuard / R8 规则

补充 `app/proguard-rules.pro`：

- 保留 Gson `TypeToken` 泛型签名。
- 保留 Xposed 入口类名，避免 `assets/xposed_init` 失效。
- 保留 Shizuku 相关 Binder / AIDL 类。
- 保留 Room 数据库、Entity、注解成员。
- 保留 kotlinx.serialization 生成的 serializer 信息。
- 对 SLF4J 可选绑定增加 `dontwarn`。

## 四、R8 启动闪退修复

### 4.1 问题现象

首次开启 R8 后，正式签名 release 包安装后启动即闪退。

通过 `adb logcat` 抓取到关键错误：

```text
TypeToken must be created with a type argument
```

崩溃发生在 Room 初始化 `Converters` 时，原因是 Gson `TypeToken` 依赖泛型签名，而 R8 默认移除了相关 `Signature` 属性。

### 4.2 修复方式

新增规则：

```proguard
-keepattributes Signature
-keepattributes RuntimeVisibleAnnotations,RuntimeInvisibleAnnotations
-keep class com.google.gson.reflect.TypeToken { *; }
-keep class * extends com.google.gson.reflect.TypeToken { *; }
```

修复后重新构建 release，并使用 debug keystore 临时签名安装验证：

- 冷启动正常。
- 无 `AndroidRuntime` 崩溃。
- App 进程正常存在。

随后正式签名包也完成烟测，未再出现启动闪退。

## 五、本地模型版分支尝试

### 5.1 分支策略尝试

曾计划同时维护两条发布线：

- `main`：标准版，发布 `Willdo`。
- `local-model/main`：本地模型版，发布 `Willdo-LocalModel`。

初始化流程：

1. 在 `main` 提交“移除标准版本地模型能力”。
2. 从新的 `main` 创建 `local-model/main`。
3. 在 `local-model/main` revert 该移除提交，恢复本地模型能力。
4. 后续通用改动由 `main` 同步到 `local-model/main`。

本地模型版开启同样 R8 / 资源压缩 / JNI 压缩后，APK 约为：

```text
24.85 MiB
```

该体积本身可以接受，但后续稳定性测试暴露了维护成本。

### 5.2 LiteRT-LM R8 混淆问题

本地模型版首次开启 R8 后，端侧推理发生 native crash。

tombstone 关键日志：

```text
JNI DETECTED ERROR IN APPLICATION: mid == null
com.google.ai.edge.litertlm.LiteRtLmJni.nativeCreateConversation
```

原因：

- R8 混淆了 `com.google.ai.edge.litertlm.*` 相关 Java/Kotlin API。
- LiteRT-LM JNI 层按原始类名 / 方法名访问，混淆后 native 找不到方法。

尝试补充 keep：

```proguard
-keep class com.google.ai.edge.litertlm.** { *; }
-keep class com.google.ai.edge.litert.** { *; }
-keepclasseswithmembernames class * {
    native <methods>;
}
-dontwarn com.google.ai.edge.litertlm.**
-dontwarn com.google.ai.edge.litert.**
```

补充后 native crash 不再复现，APK 体积仅小幅增加至约：

```text
24.93 MiB
```

### 5.3 本地模型 prepare / 前台服务问题

本轮还尝试优化本地模型 prompt 路由：

- 本地截图识别先跑 ML Kit OCR。
- OCR 文本复用 `InstantCodeParser` / `InstantCodePatterns` 判断 prompt 类型。
- 命中取件、取餐、取票、寄件时使用码类 prompt。
- 否则使用普通日程 prompt。
- 多模态本地模型仍传原图，只是 prompt 由 OCR 路由决定。
- OCR 为空且本地多模态可用时，使用统一多模态 prompt 兜底。

为了节省等待时间，曾尝试让模型 `prepare()` 和 OCR 并行。但测试中发现：

- 文本路径可能卡在 `prepare(false)` 阶段。
- 多模态路径仍可能出现前台服务超时。
- 日志出现 `Stop FGS timeout: AiEngineService`。
- 没有稳定获得 `raw response / clean json / parse status`。

初步判断问题不在 prompt，而在 `ai-engine` 前台服务生命周期、Android / OPlus 对 FGS 的限制、LiteRT-LM 运行状态等复杂因素。

### 5.4 暂停本地模型版维护

综合考虑：

- 本地模型用户使用量较少。
- 每次发布需要额外验证模型导入、模型加载、文本推理、OCR 推理、多模态推理、取消通知、前台服务等链路。
- 不同 ROM 对前台服务和 native 推理的限制差异较大。
- 标准版体积已经压缩到约 14 MiB。

最终决定：

- 暂停维护和发布 `Willdo-LocalModel`。
- `local-model/main` 分支暂时保留，作为历史排查分支。
- 后续仅维护 `main` 标准版。

## 六、本地模型残留清理提示

### 6.1 背景

由于标准版和本地模型版使用同一个 `applicationId`：

```text
com.antgskds.calendarassistant
```

用户可以直接从 LM 版覆盖安装标准版。覆盖安装后：

- 应用数据不会被清空。
- 历史本地模型文件仍留在私有目录。
- 标准版不会使用这些模型文件。

如果不处理，用户切换到标准版后仍可能占用较多存储空间。

### 6.2 实现方案

新增：

- `LocalModelResidueCenter`

职责：

- 检测 `filesDir/local_models` 是否存在残留模型文件。
- 统计文件数量和体积。
- 计算目录 fingerprint，避免同一批文件每次启动都提示。
- 暴露 pending prompt 给 UI。
- 执行清理，并关闭旧设置中的本地推理状态。

UI 接入：

- `MainActivity` 收集 `pendingPrompt`。
- 使用现有 `PredictiveFloatingActionCard` 展示。
- 标题：`检测到本地模型文件`。
- 内容展示占用大小，并说明标准版不会使用。
- 按钮：`清理` / `稍后`。

清理行为：

- 删除 `filesDir/local_models`。
- 设置：`isLocalSemanticEnabled = false`。
- 设置：`selectedLocalModelId = ""`。

提示优先级：

1. 崩溃恢复提示。
2. 异常数据清理提示。
3. 本地模型残留清理提示。
4. 剪贴板码类识别提示。
5. Prompt 更新提示。

## 七、模型配置页和日程类型 Chip 调整

### 7.1 模型配置页提示文案

模型配置页的自定义服务商需要两步操作：

1. 点击右下角按钮测试连接并拉取模型列表。
2. 选择模型后再次点击右下角按钮保存配置。

为了减少用户误解，提示文案改为根据状态变化：

- 未拉取模型：`点击右下角按钮测试连接并拉取模型列表`
- 已拉取但未选模型：`请选择模型名称，然后再次点击右下角按钮保存配置`
- 已选择模型：`再次点击右下角按钮保存配置`
- 非自定义服务商：`点击右下角按钮保存配置`

提示区从表单卡片下方拉开距离，最终使用 `120.dp` 的间距，避免过于贴近卡片。

### 7.2 新建日程类型 Chip 宽度统一

新建 / 编辑日程 Dialog 的类型选择行已改为分页式横滑。

本轮修复：

- 未展开的当前类型 Chip 宽度与展开后的 4 等分 Chip 保持一致。
- 避免点击展开前后 Chip 宽度跳变。
- 当前类型在展开后仍位于同一行，不另起一行。

## 九、当前最终决策

后续只维护标准版 `main`。

当前策略：

- 不再发布 `Willdo-LocalModel`。
- 不再继续修复本地模型推理链路。
- `local-model/main` 暂时保留，不删除，便于未来排查历史实现。
- `main` 保留本地模型残留清理提示，服务从 LM 版覆盖安装到标准版的用户。
- 标准版继续保留 R8 / 资源压缩 / JNI 压缩。

## 十、已知风险与后续待办

1. R8 规则仍需随新依赖和反射用法持续补充。
2. Xposed / HyperOS 胶囊依赖类名和外部框架，后续发版前仍需真机验证。
3. Shizuku / Root 剪贴板后台监听依赖 ROM logcat 行为，不保证所有设备稳定。
4. JNI 压缩降低 APK 下载体积，但安装后系统可能解压 `.so`，安装占用不一定同比例下降。
5. 老 LM 用户覆盖标准版后的模型残留清理提示需要继续观察真实升级场景。
6. 如果未来重新启用端侧模型，需要重新设计 `ai-engine` 前台服务生命周期，而不是只补 R8 规则。

---

## 十一、阶段结论

本轮最重要的结论是：标准版 APK 可以在不牺牲核心功能的前提下压缩到约 14 MiB。

本地模型版在体积上已不再是主要问题，但运行链路涉及 LiteRT-LM、native JNI、前台服务、ROM 限制和 R8 规则，维护成本偏高。考虑到用户使用量和发布测试成本，短期暂停本地模型版维护是更稳妥的选择。

后续开发应回到标准版主线，优先关注云端识别、日程体验、通知胶囊、备份迁移和真机稳定性。
