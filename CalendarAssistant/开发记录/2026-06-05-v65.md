## 二、问题背景

用户反馈天气预警通知的 icon 看起来统一都是 `///`。

排查后确认该现象主要来自实况胶囊路径：

1. `WeatherNotifier` 在开启实况通知时调用 `CapsuleCenter.showWeatherAlert()` 或 `showWeatherRisk()`。
2. `CapsuleStateManager` 将官方预警和风险提醒都归类为 `TYPE_WEATHER_ALERT`。
3. `IconUtils.getSmallIconForCapsule()` 对 `TYPE_WEATHER_ALERT` 固定返回 `R.drawable.ic_weather_rain_heavy`。
4. `ic_weather_rain_heavy.xml` 的视觉就是多条斜线，因此所有天气预警看起来都像 `///`。

普通天气通知路径也存在类似问题：

- 未开启实况通知时，`WeatherNotifier` 统一使用 `ic_notification_small`。
- 即使天气内容是高温、雷暴、大风、雾霾或降雪，也不会反映到通知图标上。

## 三、天气预警图标映射

### 3.1 新增统一映射器

新增 `WeatherAlertIconMapper`，专门处理天气预警和天气风险提醒的图标选择。

提供三个入口：

- `officialIconRes(alert: WeatherAlertData)`
- `riskIconRes(risk: WeatherRiskAlert)`
- `iconRes(vararg values: String)`

设计原因：

- 官方预警和风险推断虽然数据结构不同，但最终都可以归约为标题、天气文本、描述和建议。
- 统一映射可以避免普通通知、原生实况胶囊、Flyme 胶囊各自维护一套分支。
- 当前天气预警数据没有稳定的跨源标准图标字段，因此先以文本关键词做轻量映射。

### 3.2 当前关键词策略

本轮先覆盖常见预警类型：

```text
高温 / 热 / 干旱       -> 晴天
冰雹                  -> 冰雹
雷 / 强对流 / 龙卷     -> 雷暴
雨夹雪 / 冻雨          -> 雨夹雪
暴雪 / 大雪 / 结冰     -> 大雪
雪 / 低温              -> 雪
暴雨 / 强降雨 / 洪水   -> 大雨
雨 / 降水 / 降雨       -> 雨
台风 / 大风 / 风       -> 大风
雾                    -> 雾
霾 / 沙 / 尘 / 能见度  -> 雾霾
阴                    -> 阴天
云                    -> 多云
```

默认回退为 `ic_weather_partly_cloudy`，避免未知预警继续落到 `ic_weather_rain_heavy`。

## 四、通知链路接入

### 4.1 普通天气通知

`WeatherNotifier` 调整为：

- 官方预警使用 `WeatherAlertIconMapper.officialIconRes(alert)`。
- 风险提醒使用 `WeatherAlertIconMapper.riskIconRes(risk)`。

这样即使用户未开启实况胶囊，天气预警通知也能显示更接近内容的图标。

### 4.2 原生实况胶囊

`IconUtils.getSmallIconForCapsule()` 调整为：

- `TYPE_WEATHER_ALERT` 不再固定返回 `ic_weather_rain_heavy`。
- 改为读取胶囊的 `title/content/description/display` 文本，再交给 `WeatherAlertIconMapper.iconRes(...)`。

这样官方预警和风险提醒都能根据实际文案选择图标。

### 4.3 Flyme 实况胶囊

`FlymeCapsuleProvider` 中天气提醒 fallback 也从固定 `ic_weather_rain_heavy` 改为走 `WeatherAlertIconMapper`。

这一步主要是防止 Flyme 自定义 RemoteViews 场景下再次回退为 `///`。

## 五、开发者通知测试面板

### 5.1 背景

天气图标属于系统通知栏和 ROM 实况通知展示问题，只靠代码审查很难确认真实效果。

因此本轮在开发者选项中新增一个 bottom sheet，用于手动触发各种普通通知和实况胶囊，方便真机验证。

入口位置：

```text
实验室 -> 开发者选项 -> 通知/胶囊测试
```

### 5.2 普通通知测试

底部弹窗提供普通通知测试项：

- 日程
- 取件
- 课程

天气普通通知测试项：

- 高温
- 雷暴
- 暴雨
- 大风
- 雾霾
- 降雪

天气测试项会构造 `WeatherAlertData`，再使用 `WeatherAlertIconMapper.officialIconRes()` 生成通知 icon，便于直接验证本轮图标映射是否生效。

### 5.3 实况胶囊测试

底部弹窗提供实况胶囊测试项：

- OCR 进度
- OCR 完成
- 模型加载
- 网速

天气实况胶囊测试项包括：

- 官方预警类：高温、雷暴、暴雨、大风、雾霾、降雪。
- 风险推断类：高温风险、降雨风险、雾霾风险。

这些测试项全部走现有 `CapsuleCenter`，不绕开业务中心层，避免 UI 直接操作 `CapsuleStateManager`。

### 5.4 状态提示与清理

底部弹窗会提示：

- 系统通知权限未开启时，普通通知可能不会显示。
- 实况胶囊开关未开启时，实况测试不会显示。

同时新增“清除测试胶囊”按钮，会清理：

- OCR 胶囊
- 模型加载胶囊
- 天气胶囊
- 网速测试值

## 六、便签表格编辑器 UI 重构

### 6.1 问题背景

本轮便签开发中，表格编辑器暴露出一组视觉和交互问题：

- 表格行/列操作点曾经被绘制进表格内部，影响单元格内容区。
- 表格右侧边框在窄屏下容易被裁掉，默认宽度和位置不够稳定。
- 行 list 与列 list 之间存在方向和布局串扰，点击一侧时另一侧会出现异常排布。
- 行/列选中时，整块 list 区域和表格外框会被一起高亮，视觉语义不清晰。
- 选中态下表格会因为行/列 list 的显隐发生横向跳动。

### 6.2 透明表格板块

最终放弃了早期 `markerOverlay + scrollX` 的悬浮坐标方案，改为透明表格板块结构：

```text
TablePanel
  -> RowHeaderColumn
  -> HorizontalScrollView
       -> TableContentColumn
            -> ColumnHeaderRow
            -> TableShell
```

设计要点：

- 左侧行 list 作为正常布局列，不再画到负坐标，避免被父容器裁剪。
- 顶部列 list 放进 `HorizontalScrollView` 内部，天然跟随表格横向滚动。
- 表格板块固定预留 `24dp` 透明 padding，用于放置当前行/列操作点和滚动安全区。
- 横向滚动继续通过自定义 `InterceptingHorizontalScrollView` 隔离父级竖向滚动，减少滑动冲突。

### 6.3 行列操作点策略

本轮没有保留“所有行列 list 全量显示”的模式，而是改为只显示当前选中单元格对应的操作点：

- 当前行显示 `⋮`。
- 当前列显示 `⋯`。
- 非当前行/列不显示操作点，降低视觉噪音。
- 移除选中行/列后出现的主题色小药丸，只保留蓝色图标和对应边框语义。

### 6.4 高亮与边框修复

表格选中高亮重新以真实 `gridContainer` 为坐标基准，避免被 `tableShell` 的 padding 顶偏。

高亮层级调整为：

- 单元格选中：只画单元格边框。
- 行选中：只画该行边框和当前行操作点。
- 列选中：只画该列边框和当前列操作点。
- 表格本体边框保持中性灰，不再因为选中单元格而整体变成主题色。

## 七、便签段落样式共存与回车逻辑

### 7.1 新增列表样式模型

旧模型中，`NoteParagraphStyle` 同时承载标题、正文、引用、代码、无序列表和有序列表。

这导致：

- H 与无序/有序列表无法共存。
- 各类块级标签会互相覆盖。
- 段首回车后，样式容易被重置为正文。

本轮新增独立列表样式：

```kotlin
enum class NoteListStyle {
    NONE,
    BULLET,
    ORDERED
}
```

`NoteParagraph` 新增字段：

```kotlin
val listStyle: NoteListStyle = NoteListStyle.NONE
```

这样标题样式与列表样式分离，支持：

- `H1/H2/... + 无序列表`
- `H1/H2/... + 有序列表`
- `TODO + listStyle`
- `H/list/TODO + 行内加粗、斜体、下划线、删除线`

### 7.2 旧数据兼容

为了兼容旧便签数据，新增迁移辅助逻辑：

- `NoteParagraph.effectiveListStyle()`
- `NoteParagraph.effectiveParagraphStyle()`
- `NoteParagraph.withMigratedParagraphStyle()`
- `NoteDocument.withMigratedParagraphStyles()`

旧数据中：

```text
style = BULLET  -> style = BODY + listStyle = BULLET
style = ORDERED -> style = BODY + listStyle = ORDERED
```

读取和保存时会自动归一，避免旧列表样式丢失。

### 7.3 段首回车修复

修复前，光标位于 H/list 段开头回车时，编辑器会把当前文本移动到下一行，并让标签丢失。

修复后：

- 光标在段首且当前行非空时，回车会在当前段前插入一个空白正文行。
- 原段文本、标题样式、列表样式、待办状态和行内样式保持不变。
- 焦点落到新插入的空白行。

同时保留列表空行退出逻辑：

- 空 TODO 行回车会退出 TODO。
- 空 list 行回车会清除 `listStyle`。

## 八、便签 Markdown 与工具栏调整

### 8.1 Markdown 导入导出

`NoteMarkdownCodec` 同步适配组合样式。

导出支持：

- 普通标题：`# 标题`
- 无序列表标题：`- # 标题`
- 有序列表标题：`1. ## 标题`
- 待办：`- [ ] 文本` / `- [x] 文本`
- 有序待办：`1. [ ] 文本`
- 表格继续使用 Markdown table 格式。

导入时会解析列表前缀、待办前缀和标题前缀，并还原到 `style + listStyle + type` 的组合结构。

### 8.2 便签工具栏颜色

便签浮动工具栏颜色向底栏风格靠齐：

- 主工具栏背景使用 `MaterialTheme.colorScheme.surface`。
- 工具项 chip 使用 `secondaryContainer`。
- 普通图标和文字使用 `onSurfaceVariant`。
- 加号主按钮继续使用 `primary / onPrimary`。

这次只调整颜色体系，不改变工具栏主布局。

### 8.3 右上角操作保留平铺

曾短暂尝试将编辑页右上角改为单个 list 图标并展开菜单，包含置顶和删除。

复测后决定退回原交互：

- 右上角继续平铺显示“置顶/取消置顶”和“删除”两个按钮。
- 置顶状态仍用主题色标识。
- 删除仍走原来的确认流程。

## 十一、阶段结论

本轮修复的核心不是单个图标资源，而是把“天气提醒是什么类型”从固定类型分支中抽离出来，统一交给天气文本映射层处理。

最终结构变为：

```text
WeatherAlertData / WeatherRiskAlert
    -> WeatherAlertIconMapper
    -> 普通通知 / 原生实况胶囊 / Flyme 胶囊
```

同时补上开发者测试 bottom sheet，后续再遇到通知栏、胶囊、天气图标类问题，可以在不伪造真实天气接口返回的情况下快速触发和验证。

便签部分的核心变化是把“段落样式”和“列表样式”解耦，并把表格操作点从悬浮坐标系统迁移到真实布局结构中。

最终便签编辑器方向变为：

```text
NoteParagraph.style      -> 标题 / 正文 / 引用 / 代码
NoteParagraph.listStyle  -> 无序 / 有序 / 无列表
NoteParagraph.type       -> TEXT / TODO / IMAGE / FILE / DIVIDER / TABLE
```

这样后续继续扩展标签共存、Markdown 导出和复杂块编辑时，不再需要靠单一枚举互相覆盖。
