## 二、悬浮窗展开方向

### 1) 问题背景

- 用户希望悬浮窗可以选择从左侧或右侧展开。
- 现有设置只有 `edgeBarSide`，语义上只适合控制“侧边栏唤起条位置”。
- 悬浮窗入口不止侧边栏，还包括长按音量+、快捷磁贴，因此展开方向需要独立配置。

### 2) 最终规则

- `floatingExpandSide`：控制悬浮窗内容面板从左侧或右侧滑入。
- `edgeBarSide`：只控制侧边栏唤起条固定在左侧或右侧。
- 快捷磁贴、长按音量+、侧边栏唤起都遵循 `floatingExpandSide`。
- 底部输入栏保持居中，不随展开方向镜像。

### 3) 落地结果

- `MySettings` 新增 `floatingExpandSide`，默认 `RIGHT`，兼容老用户。
- 设置页新增“悬浮窗展开方向”。
- 悬浮窗主面板根据方向切换滑入/滑出动画、对齐方式和卡片贴边 padding。
- 左侧展开时，日程卡片与便签卡片交互镜像：
  - 右滑露出编辑/状态按钮。
  - 继续右滑触发完成/恢复/签到等状态动作。
  - 左滑触发归档或删除飞出。
- 右侧展开保持原有交互不变。

## 三、便签编辑状态一致性

### 1) 问题现象

- 新建便签后修改内容，返回列表再进入，偶尔显示修改前内容。
- 再返回再进入，又可能显示修改后的内容，表现为新旧状态交替。

### 2) 根因判断

- `NoteEditorScreen` 的草稿状态按 `initialNote?.id` 记忆。
- 同一条便签修改前后 id 不变，导致编辑器可能复用旧草稿。
- `HomeScreen` 中 `noteEditorInitialNote` 依赖异步/动画时机，容易在退出与再次进入之间拿到旧对象。
- 保存后列表如果尚未刷新，再次点入也可能拿到旧事件对象。

### 3) 落地结果

- `HomeScreen` 新增 `noteEditorSessionKey`，每次打开便签编辑器都会生成新 session。
- 新建/编辑便签时同步设置 `noteEditorInitialNote`，不再依赖延迟同步。
- `NoteEditorScreen` 的标题、正文、编辑器 controller 按 session 初始化。
- `BlockNoteEditor` 外层使用 `key(editorSessionKey)`，避免正文块编辑器复用上一轮内部状态。
- `ScheduleCenter.updateEvent` 在写库前先乐观更新 `_events`，降低保存后马上返回列表时点到旧数据的概率。

**关键文件**

- `app/src/main/java/com/antgskds/calendarassistant/ui/page_display/HomeScreen.kt`
- `app/src/main/java/com/antgskds/calendarassistant/ui/page_display/NoteEditorScreen.kt`

## 四、便签光标偶发难呼出排查

### 1) 排查结论

- 当前 `NoteBlockEditor.kt` 已包含 Claude 对焦点/点击竞争问题的核心修复：
  - 外层使用 `.clickable`，避免父级 `pointerInput/detectTapGestures` 与子输入框抢事件。
  - `BlockEditorField` 使用 `withFrameNanos {}` 后再 `requestFocus()`。
  - `beginEdit` 已处理 id 漂移。
  - `commitActive` 使用现场 parse，避免闭包中的旧 blocks。
  - `SideEffect` 注册 controller 闭包，保证每次 recomposition 拿到最新状态。
- 本地旧分支 `feature/note-editor-ai-keep-open` 是更早的实现，不符合 Claude 最终修复清单，不建议合回。

### 2) 本轮决策

- 当前问题表现为偶发，且用户复测时恢复正常。
- 本轮不改 `NoteBlockEditor.kt`，避免对已修过的焦点链路引入新风险。
- 如果后续复现频率升高，再考虑给 `BlockEditorField` 增加一次短延迟 retry focus/show keyboard 的兜底。

## 五、识别失败胶囊稳定展示

### 1) 问题现象

- 有时识别失败后，“正在分析/正在识别”胶囊直接消失。
- 期望行为是展示“识别失败”胶囊，并给出可理解的失败原因。

### 2) 根因

- `RecognitionCenter.analyzeImage()` 失败时会发出 `RECOGNITION_FAILED` 事件。
- `TextAccessibilityService` 订阅失败事件后，会先取消进度胶囊，再显示失败结果胶囊。
- 但 `processScreenshot()` 在 `analyzeImage()` 返回后又会进入主线程块，并先无条件执行 `cancelProgressNotification()`。
- 如果失败事件先显示了失败胶囊，随后这个无条件 cancel 会把失败胶囊清掉，形成竞态。

### 3) 落地结果

- `AnalysisResult.Empty` / `AnalysisResult.Failure` 返回后，不再由 `processScreenshot()` 再次清理胶囊。
- 失败展示统一交给 `RECOGNITION_FAILED` 事件订阅处理。
- `Bitmap.wrapHardwareBuffer(...) == null` 不再静默清掉进度胶囊，改为显示“截图处理失败 / 请重试”。
- 空 Bitmap 分支补充关闭 `hardwareBuffer`，避免异常路径资源泄漏。

**关键文件**

- `app/src/main/java/com/antgskds/calendarassistant/service/accessibility/TextAccessibilityService.kt`

## 六、识别失败文案与错误码展示

### 1) 最终展示规则

- 不向普通用户展示 `ANALYSIS_FAILURE`、`EMPTY_RESULT`、`BITMAP_NULL` 这类内部枚举。
- 优先展示已有中文原因，例如“网络超时”“密钥无效(401)”“请求过多(429)”。
- HTTP 类错误继续保留状态码，方便用户和开发者定位配置/接口问题。
- 系统截图数字错误码映射为中文，不直接展示数字。

### 2) 当前映射

- `EMPTY_RESULT`：`未识别到有效日程`
- `ANALYSIS_FAILURE`：优先显示具体中文原因；无具体原因时显示 `分析失败，请稍后重试`
- `BITMAP_NULL`：`截图处理失败 / 请重试`
- `ERROR_TAKE_SCREENSHOT_INTERVAL_TIME_SHORT`：`截图太频繁，请稍后再试`
- `ERROR_TAKE_SCREENSHOT_NO_ACCESSIBILITY_ACCESS`：`截图权限不可用，请检查无障碍服务`
- `ERROR_TAKE_SCREENSHOT_INVALID_DISPLAY`：`当前屏幕不可截图，请重试`
- `ERROR_TAKE_SCREENSHOT_INVALID_WINDOW`：`当前窗口不可截图，请重试`
- `ERROR_TAKE_SCREENSHOT_SECURE_WINDOW`：`当前页面禁止截图`
- `ERROR_TAKE_SCREENSHOT_INTERNAL_ERROR`：`系统截图失败，请重试`

### 3) 调试信息

- 系统截图失败的原始数字错误码仍写入日志：
  `takeScreenshot 失败: code=<errorCode>`
- 用户侧只看到中文提示。
