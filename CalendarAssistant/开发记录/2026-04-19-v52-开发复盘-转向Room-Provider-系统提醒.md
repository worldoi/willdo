# 2026-04-19 v52 开发复盘：转向 Room <-> Provider <-> 系统提醒

## 本次开发背景

- 目标原本是硬切到 provider-only：应用不再依赖本地持久层，直接以 Calendar Provider 为唯一数据源。
- 同时推进了课程并入普通日程（`tag=course`）与重复日程 `THIS` 语义操作。
- 在设备实测阶段暴露了关键稳定性问题，最终决定调整架构方向。

## 本次实际做过的改造（按阶段）

1. 推进了 provider-only 方向：
   - 新增/迁入 center-port-node 与 provider 读写节点。
   - 将部分同步链路改为直接从 Provider 回流。
   - 清理了部分旧同步组件。

   - 普通日程、重复日程创建失败（对用户表现为弹窗消失但事件未创建）。
   - 查看/编辑部分系统重复日程时提示“未找到对应的重复系列信息”。
3. 进行止血：
   - 回到稳定 git 快照（Room 主链路版本）。
   - 恢复开发记录文档（从备份 zip 提取缺失 md）。

## 暴露出来的核心问题

### 1) tag 写入耦合事件写入，导致整批回滚

- `tag` 依赖 `CalendarContract.ExtendedProperties`。
- 在非本地账号日历上，`ExtendedProperties` 可能被系统/账户策略拒绝。
- 当 `Events + Reminders + ExtendedProperties` 同批提交时，任一步失败会导致整批失败，最终“事件根本没创建出来”。

### 2) provider-only 下的重复实例语义不完整

- UI/逻辑中仍存在“依赖本地 parent 记录”的路径。
- 但 Provider 查询常常给的是实例视图，不保证本地有对应 parent 对象，导致 `THIS/FUTURE/ALL` 的入口被误拦截。

### 3) 前端状态无独立落点时，业务动作可恢复性差

- 归档、已完成、已检票、撤销窗口（Undo）等并非 Calendar 标准字段。
- 纯 Provider 模式下，这些语义要么丢失，要么被迫塞到不可靠扩展字段，维护成本和故障概率都高。

## 最终决策：转向 Room <-> Provider <-> 系统提醒

### 架构角色划分

- `Provider`：日程事实源（时间、重复规则、实例操作）。
- `Room`：业务元数据与前端状态源（tag、归档、完成态、检票态、Undo 等）。
- `系统提醒`：提醒调度执行层（与事件事实和前端状态联动）。

### 数据流原则

1. **写入主路径**：先写 Provider（失败即失败），再更新 Room 投影与状态。
2. **tag 策略**：
   - 可尝试写 Provider 扩展字段，但必须是 best-effort；
   - 失败不允许回滚事件主写入；
   - Room 持久保存 tag 作为稳定兜底。
3. **读取主路径**：Provider 实例流 + Room 状态合并渲染。
4. **重复实例语义**：`THIS/FUTURE/ALL` 以 Provider 语义为准，Room 仅维护业务状态映射，不反客为主。

## 为什么最终选择这条路

- 能同时满足三个刚性要求：
  1) 非本地日历可用（保留 `update this` 能力）；
  2) tag/业务态可持续（不依赖不稳定扩展写入）；
  3) 提醒与撤销语义可控（系统提醒层有稳定落点）。
- 对用户体验最关键的“能创建、能编辑、能回显”优先恢复。
- 与参考项目（Simple Calendar）一致：业务语义由本地模型承载，Provider 负责系统日程互通。

## 当前状态与后续

- 当前仓库已恢复到稳定快照并可编译、可装机。
- 后续按“Room <-> Provider <-> 系统提醒”重构时，优先顺序：
  1. 恢复创建链路稳定性（先事件后元数据，失败可见）。
  2. 修正重复实例入口判定（去掉对本地 parent 的硬依赖）。
  3. 建立 Room 元数据表与 Provider 实例的稳定映射。
  4. 回归测试：新建、重复编辑、系统外部变更回流、归档/完成/撤销、提醒触发。
