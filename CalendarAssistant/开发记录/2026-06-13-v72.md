## 二、天气通知地点名兜底

### 2.1 问题现象

用户反馈天气预警胶囊通知中偶发显示：

```text
当前位置 · 蓝色海上大风预警
当前位置 · 黄色雷电预警
```

这类文案对用户没有实际定位信息，也会让通知看起来像定位失败。

### 2.2 根因分析

天气刷新链路中，自动定位先拿到经纬度，再通过和风 `/geo/v2/city/lookup` 做地点补全。

偶发显示“当前位置”的原因不是预警接口本身错误，而是地点补全链路可能失败或不稳定：

- Android 定位来源会在 GPS、网络、缓存、被动定位之间切换，坐标可能有轻微漂移。
- 地理反查与天气/预警接口是独立请求，反查失败时仍可能成功拿到天气与预警。
- 旧逻辑在反查失败时会把空地点兜底成 `当前位置`，并继续传给通知模板。
- 原先还会在补全前缓存原始坐标，导致缓存中可能保留没有城市名的“半成品位置”。

### 2.3 修复方案

本轮调整 `WeatherRepository` 的位置处理策略：

- 自动定位成功后不再立即缓存原始坐标。
- 先执行地点补全，再写入普通缓存。
- 只有带可信名称的信息才写入“可信位置缓存”。
- 当本次反查失败但当前位置距离上一次可信位置在 10km 内时，复用上一次可信城市名、行政区和 locationId。
- 可信位置缓存最长保留 30 天，避免长期使用过期地点名。

### 2.4 通知层防御

通知展示层增加二次过滤：

- `WeatherData.displayNotificationLocationName()` 不再把 `当前位置`、`最近位置`、`手动位置` 作为通知地点名。
- `NotificationTemplateCenter.compactLocationName()` 同样拒绝这三个泛化地点名。
- 如果最终仍没有可信地点名，官方天气预警仍会发送，但标题中不再拼接 `当前位置 ·` 前缀。

这样既避免误导用户，也不会因为地点名缺失而吞掉重要预警。

## 三、剪贴板识别可靠性补丁

### 3.1 问题现象

历史反馈中存在“打开软件后，剪贴板中的取件码偶发没有弹出识别卡”的情况。

复盘后确认问题更偏向触发时机和状态可靠性，而不是取件码正则本身：

- `onResume()` 触发可能早于窗口真正获得焦点，Android 10+ 后台剪贴板读取容易失败。
- 有 Shizuku/Root 权限时，前台确认卡逻辑直接跳过，但后台 logcat 监听也可能错过当前剪贴板。
- 自动入库失败或返回空结果时，旧逻辑仍会把该内容标记为已处理，导致后续 10 分钟内不再提示。
- 前台确认卡和后台自动入库可能并发处理同一份剪贴板，存在竞态风险。

### 3.2 前台触发改造

`MainActivity` 中剪贴板前台兜底检查从 `onResume()` 移到：

```kotlin
onWindowFocusChanged(hasFocus = true)
```

这样实际读取发生在 Activity 获得窗口焦点之后，更符合 Android 剪贴板访问限制。

### 3.3 Center 状态机

`ClipboardCodeCenter` 新增一组本地运行态：

- `processingKeys`：正在处理的剪贴板实例或业务指纹，避免前台/后台竞态。
- `handledKeys`：成功处理记录，保留 10 分钟。
- `ignoredKeys`：用户取消记录，保留 10 分钟。
- `failedKeys`：失败短冷却，当前 15 秒。
- `contentInstances`：用 `SystemClock.elapsedRealtime()` 生成本地剪贴板实例 key，不依赖 OEM 可能漂移的 `ClipDescription.timestamp`。
- `lastSeenPromptTextHash` / `lastPromptedTextHash`：记录当前剪贴板内容是否已经弹过前台确认卡，避免窗口重新获得焦点时对同一内容反复提示。

同一内容在短窗口内复用同一个实例 key，避免每次读取都生成新 key；同一业务取件码还会继续通过 `SmsPickupFingerprint` 做跨文案去重。

### 3.4 成功/失败语义修正

本轮修正旧逻辑中最关键的误判：

- `confirmPendingPrompt()` 只有 `ingestInstantCode(...)` 返回非空结果时才标记 handled。
- `autoIngestText()` / `autoIngestCurrentClipboard()` 只有实际入库成功才标记 handled。
- 入库失败或返回空结果时只进入 15 秒 failed cooldown，不再压制 10 分钟。

### 3.5 重复提示抑制

由于前台检查移动到 `onWindowFocusChanged(hasFocus = true)`，打开 App 或从系统界面返回 App 时会再次读取剪贴板。为避免剪贴板仍是同一个取件码时每次都弹入库卡，`ClipboardCodeCenter` 增加同内容提示锁：

- 剪贴板内容 hash 变化时，重置本轮提示状态。
- 同一内容第一次识别到取件码时正常弹出确认卡。
- 同一内容已经弹过且剪贴板没有变化时，后续窗口焦点触发直接跳过。
- 用户复制新的取件码或其它内容后，再次检测到新取件码仍会正常提示。

这样保留“无 Shizuku/Root 时打开 App 检查剪贴板”的入口，同时避免同一剪贴板内容在本次进程内反复打扰。

### 3.6 后台监听联动

`ClipboardCodeMonitorService` 现在会向 `ClipboardCodeCenter` 同步后台监听状态：

- 服务启动后设置 `privilegedMonitorActive = true`。
- 服务销毁、权限丢失或循环退出时设置为 `false`。
- 当前台检测发现 Shizuku/Root 后台监听正在活跃时，会静默跳过，避免与后台自动入库竞争。
- 后台监听每次启动或重启 logcat 监听前，主动执行一次 `autoIngestCurrentClipboard("clipboard_monitor_start")`，补上“服务启动时剪贴板已经存在内容”的场景。

## 四、悬浮确认卡层级修复

### 4.1 问题现象

用户反馈 `FloatingActionCard` 层级异常，重点提到取件码入库确认卡会被首页底部悬浮栏压住；此前也排查过删除随口记、删除日程等确认卡层级。

排查发现：

- 首页底部 `IntegratedFloatingBar` 使用了 `zIndex(3f)`。
- 早期尝试让 `PredictiveFloatingActionCard` 向 `PredictiveBottomDialogHost` 传入 `.zIndex(10f)`，但该方案仍依赖同一 Compose 父层级内的绘制顺序。
- `PredictiveBottomDialogHost` 内部根 `Box` 还曾写死为 `Modifier.fillMaxSize()`，没有使用外部传入的 `modifier`，导致第一轮层级修复未生效。
- 删除随口记、删除便签、删除日程、重复日程删除等确认卡在 Material3 首页内已经位于 `HomeScreen` 底部栏之后的同层 overlay 区域。
- 取件码入库卡此前在 `MainActivity` 全局弹窗区渲染，和首页删除卡不是同一层。

卸载重装 debug 包后问题仍复现，最终确认这不是缓存问题。继续依赖通用组件内部 `zIndex` 或 Dialog 化都会扩大影响面；更合适的修法是把首页取件码入库卡移动到和删除卡相同的 `HomeScreen` overlay 层。

### 4.2 组件层修复

本轮最终收窄为首页调用点修复：

- `PredictiveFloatingActionCard` 回退为普通 `PredictiveBottomDialogHost` 承载，不再依赖 `.zIndex(10f)`，也不再改为平台级 Dialog。
- `PredictiveBottomDialogHost` 保留公共 `modifier` 透传修复。
- `MainActivity` 将取件码 `clipboardPrompt` 在 Material3 首页时传入 `HomeScreen`，并停用原全局取件码卡，避免重复显示。
- `HomeScreen` 在 `IntegratedFloatingBar` 之后、删除卡之前渲染取件码入库卡，复用删除卡同一层级与同一底部偏移。
- 非首页或 MIUI 首页仍使用 `MainActivity` 原全局取件码卡，避免影响其它入口。

保留的公共 Host 修复：

```kotlin
Box(modifier = modifier.fillMaxSize())
```

这样首页取件码入库卡不再跨层级渲染，而是和以下首页确认卡处于同一 overlay 层：

- 删除随口记
- 删除便签
- 删除日程
- 删除重复日程
- 编辑重复日程确认
- 其他设置页/备份页使用的预测返回悬浮确认卡

## 五、悬浮窗随口记播放按钮色彩修正

### 5.1 问题现象

用户反馈深色模式下悬浮窗单条随口记卡片中的播放按钮过亮，视觉上比底部输入框和卡片主体更抢眼。

排查发现按钮容器使用固定浅蓝色：

```kotlin
Color(0xFFEAF1FF)
```

该颜色不会随主题深浅动态调整，在深色模式下显得突兀。

### 5.2 修复方案

本轮参考悬浮窗底部输入框中“随口记/日程”切换按钮的状态样式，改为：

- 普通播放按钮容器使用 `MaterialTheme.colorScheme.primary.copy(alpha = 0.15f)`。
- 播放图标继续使用 `MaterialTheme.colorScheme.primary`。
- 已完成随口记保留灰态逻辑，不与普通状态混淆。
- 移除固定浅蓝色常量，避免深色模式和动态主题下产生色彩割裂。

该调整位于 `FloatingScheduleScreen`，同时影响悬浮窗单条语音随口记的默认播放按钮样式。

## 六、随口记页与详情页 UI 打磨

### 6.1 设计方向

本轮参考 ColorOS 随口记的排版思路，但没有照搬其视觉素材和功能入口。

最终方向：

- 随口记页不加入筛选 pill，继续保持单一记录流。
- 用颜色标签/左侧色条区分普通随口记、待办随口记和已完成状态。
- 语音和文本作为内容形态，不作为顶层分类。
- 详情页使用应用内统一的 `Scaffold + CenterAlignedTopAppBar`，只保留系统导航栏安全区，不加入首页底部悬浮导航。
- 不加入 ColorOS 的“问一问”浮动入口，避免 UI 先于真实功能出现。

### 6.2 列表页文档流改造

`QuickMemoPage` 从原先偏卡片/气泡的展示，调整为更轻的文档流：

- 按日期分组显示，分组标题使用“今天 / 昨天 / M月d日 EEEE”。
- 列表项去掉外层卡片背景，改为透明内容流。
- 左侧色条放大，用于表达普通、待办、已完成等状态。
- 语音随口记首行展示波形和右侧固定播放按钮，正文与时间信息放在下方。
- 文本随口记独立使用“左色条 + 正文/时间列”的布局，不再占用空波形行。
- 正文宽度不再被旧的最大宽度过早限制，尽量延展到右侧播放按钮安全距离前。

### 6.3 语音波形与 UI 大小适配

随口记页和详情页新增 `uiSize` 透传，跟随应用 UI 大小设置调整关键尺寸：

- 列表左右留白、条目间距、色条尺寸。
- 正文字号与行高。
- 播放按钮与图标尺寸。
- 详情页顶部波形、正文、待办按钮和推荐项尺寸。

语音波形宽度改为根据 `QuickMemoEntity.audioDurationMs` 动态计算，采用最小/最大宽度与对数曲线映射，避免短语音过长、长语音无限扩张。

### 6.4 详情页沉浸式编辑

`QuickMemoDetailPage` 调整为应用内统一详情页结构：

- 顶部使用 `CenterAlignedTopAppBar` 和标准返回按钮。
- 内容区使用透明背景与留白排版，不再堆叠大卡片。
- 语音详情页顶部展示居中的波形与右侧播放按钮，移除胶囊背景。
- 正文改为直接 `BasicTextField` 编辑，保留空态提示“点击输入正文...”。
- 点击详情页正文输入框外的空白区域会清除焦点，收起光标。
- 待办完成按钮改为轻量胶囊样式，减少对正文内容的抢占。

### 6.5 日程待办推荐样式

详情页中的“日程待办”推荐项参考 All Events 页面视觉：

- 推荐项保持透明背景，不使用重卡片。
- 左侧使用细色条强调候选日程。
- 标题加粗，时间信息使用主色。
- 右侧“添加”使用浅主色胶囊按钮，和当前应用按钮风格保持一致。

### 6.6 便签入口收敛

随口记页顶部暂时移除“便签”跳转按钮。

该改动只隐藏入口，不删除便签模块。后续便签会作为独立 App 方向处理，避免当前随口记页继续承担便签切换入口。

## 七、随口记备份接入

### 7.1 接入范围

本轮补齐数据备份中随口记相关内容，但不接入便签备份。

原因是便签模块后续会从当前软件内移除，不再继续扩大当前全量备份对便签模块的依赖。

本次备份范围限定为：

- 随口记正文与类型。
- 转写、分析、待办等状态。
- 随口记日程建议数据。
- 语音随口记录音文件。

ASR 模型文件不纳入备份，避免备份包体积过大，同时保留用户自行替换模型的灵活性。

### 7.2 备份结构升级

`AppBackupModels` 将备份结构升级到 `version = 2`，新增：

- `includeQuickMemos`：控制是否导入/导出随口记。
- `AppBackupQuickMemoDto`：记录随口记正文、类型、音频文件名、状态、时间和排序信息。
- `AppBackupQuickMemoSuggestionDto`：记录随口记日程建议。
- `quickMemosImported`：用于导入完成提示。

旧备份依旧通过默认值和 `ignoreUnknownKeys` 兼容读取。

### 7.3 录音文件导出与恢复

随口记录音无法放入普通 JSON，因此勾选“随口记”时导出自动走 ZIP。

ZIP 中新增目录：

```text
quick_memos/audio/
```

导入时会将录音文件复制到当前设备的：

```text
filesDir/quick_memos/audio/
```

并重写 `QuickMemoEntity.audioPath`，避免跨设备恢复后仍指向旧设备绝对路径。

### 7.4 追加导入与去重

随口记导入采用追加策略，不覆盖当前设备已有数据。

去重 key 使用：

```text
type + createdAt + bodyText + audioDurationMs
```

这样重复导入同一个备份包时不会明显产生重复随口记。

随口记建议中的旧 `eventId` 不跨设备复用，导入时清空并在原本有关联事件的情况下回退为 `PENDING`，避免错误关联到当前设备上的其它日程。

### 7.5 UI 文案

数据备份页新增“随口记”选项。

底部选项说明最终收敛为：

```text
包含随口记、状态和录音
```

避免在选项说明中暴露 ZIP 等实现细节。
