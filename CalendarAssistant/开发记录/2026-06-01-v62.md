## 二、便签重写背景

旧版便签长期复用日程 `Event` 模型：

- 标题存 `Event.title`。
- 正文存 `Event.description`。
- 通过 `tag = note / 便签` 区分便签。
- 悬浮窗待办依赖 Markdown 形式的 `- [ ]` / `- [x]`。
- 便签图片、附件、待办和正文都被迫混在日程链路中。

这种方案的核心问题是：便签和日程的生命周期完全不同，但被塞进了同一个模型。

直接后果：

1. 日程列表、归档、小组件、胶囊、系统日历同步都需要额外过滤 note tag。
2. 便签被迫带上 `startTS/endTS/state/reminder/source/importId` 等日程字段。
3. 系统日历同步链路需要不断避免把便签同步到 Calendar Provider。
4. 旧 Markdown 文本同时承担存储、渲染、编辑和待办状态，边界混乱。
5. 图片、附件、待办如果继续塞入 Markdown，光标、回车、删除、导入导出都会变得脆弱。

因此本轮最终决定：新版便签独立为 `notes` 表，不再复用 `events.description`。

## 三、为什么放弃 Markdown

本轮围绕便签编辑器做过多轮尝试，最终明确放弃“以 Markdown 作为日常编辑主模型”。

### 3.1 双排版引擎导致编辑态与渲染态割裂

旧版和早期实验版中，便签正文大致是：

```text
渲染态：AndroidView + TextView + Markwon
编辑态：Compose BasicTextField
工具栏：拼接或替换 Markdown 前缀
```

这导致一系列问题：

- 同一段文字在渲染态和编辑态的换行不同。
- 点击渲染文字后，进入编辑态时光标位置不一致。
- 长文本如 `111111...` 在显示和编辑时排版不一致。
- 点击工具栏后，光标可能跳到行首。
- `- # 标题` 这类组合语法会被错误解析或渲染出异常样式。

根因是两套排版引擎无法保证完全一致，并且 Markdown 原文和渲染后的 UI 不是一一对应关系。

### 3.2 Markdown 不适合作为本应用的移动端主编辑模型

Markdown 适合作为导入/导出格式，但不适合作为 Will do 便签的主编辑模型。

原因：

- 用户不希望日常编辑时看到 `#`、`- [ ]`、`>`、代码围栏等语法标记。
- 待办应该是可点击 checkbox，而不是字符串。
- 图片和附件应该是块或卡片，而不是 Markdown 文本里的路径。
- 删除图片、附件、待办框时，用户期待删除一个 UI 对象，而不是编辑一段语法。
- 如果为了兼容完整 Markdown 做 AST、嵌套列表、跨块选区和反向映射，会变成专业编辑器工程，偏离 Will do 的产品核心。

### 3.3 尝试过但放弃的方案

#### 方案一：继续修旧 Markdown 编辑器

尝试内容：

- 调整编辑态 padding、背景、字号和行高。
- 修工具栏组合语法。
- 避免 `- #`、`# -` 等错误拼接。

放弃原因：

- 只能缓解，不能解决双排版引擎割裂。
- 光标点击位置和显示位置仍可能不一致。
- 待办、图片、附件仍然要靠 Markdown 字符承载。

#### 方案二：单大输入框 + Markdown 语义解析

尝试内容：

- 使用一个大 `BasicTextField(TextFieldValue)` 作为唯一输入源。
- 额外写语义解析层，理解 `- # 标题` 是“列表里的标题”。
- 工具栏根据当前光标所在行做结构化改写。

放弃原因：

- 技术上可行，但用户仍然看到 Markdown 标记。
- 视觉效果不像普通便签，仍像 Markdown 编辑器。
- 用户明确反馈效果不接受。

#### 方案三：块编辑器 / 多 BasicTextField

尝试内容：

- 每个块一个 `BasicTextField`。
- 待办、标题、文本块分开渲染。
- 接近 Notion/Craft 类块编辑器思路。

放弃原因：

- 回车后光标消失。
- 跨块退格、合并、删除需要大量手写逻辑。
- 多输入框切碎了 Android 原生输入法、光标和选区系统。
- 焦点随 Compose 重组丢失，用户体验不可接受。

#### 方案四：先做纯文本，后续再加待办和附件

尝试内容：

- 第一版只保留标题、正文、自动保存、AI 分析。
- 暂时移除待办、图片、附件、格式。

放弃原因：

- 短期能稳定，但回避不了未来问题。
- 一旦后续加入图片、附件、待办，还是会遇到“特殊内容和文本混排”的同类问题。

### 3.4 最终选择

最终采用 OPPO 便签、MiNotes、Techo 等思路的折中方案：

```text
一篇便签 = 标题 + 多个逻辑行/内容块

普通行 = EditText
待办行 = CheckBox + EditText
图片行 = 图片预览块
文件行 = 文件卡片块
```

这里的“行”是用户按回车分隔的逻辑行，不是屏幕自动换行后的视觉行。

## 四、新版便签数据模型

### 4.1 独立 notes 表

新增独立 Room 表与数据层：

- `NoteEntity`
- `NotesDao`
- `NoteRepository`
- `NoteCenter`

数据库仍复用 `events.db`，但便签不再写入 `events` 表。

`EventsDatabase` 升级到 version 6，新增 `notes` 表。

### 4.2 NoteDocument

新版便签使用结构化文档模型：

- `NoteDocument`
- `NoteParagraph`
- `NoteParagraphType`
- `NoteParagraphStyle`
- `NoteTextSpan`

每个 `NoteParagraph` 是一个逻辑行/块。

段落类型：

```text
TEXT   普通文本行
TODO   待办行
IMAGE  图片行
FILE   文件附件行
```

段落样式：

```text
BODY
H1 / H2 / H3 / H4 / H5
QUOTE
CODE
BULLET
ORDERED
```

文本样式：

```text
BOLD
ITALIC
UNDERLINE
STRIKE
```

附件字段：

```text
attachmentPath
attachmentName
attachmentMime
```

这让图片和附件成为结构化块，而不是 Markdown 字符串。

## 五、编辑器最终方案

### 5.1 原生行编辑器

最终 `PlainNoteEditor` 使用 `AndroidView` 承载原生 View，内部是 `NativeLineNoteEditor`。

渲染规则：

- 普通文本行：原生 `EditText`。
- 待办行：`CheckBox + EditText`。
- 图片行：`ImageView` 预览。
- 文件行：圆角文件卡片。

这样可以把输入法、光标、选区、回车、删除交给 Android 原生控件处理，避免 Compose 多 `BasicTextField` 的焦点和重组问题。

### 5.2 待办实现

待办不再是 Markdown，而是 `NoteParagraph.type = TODO`。

规则：

1. 工具栏点击“待办”，当前逻辑行变为待办行。
2. 勾选状态保存在 `NoteParagraph.checked`。
3. 长文本自动换行仍属于同一个待办行。
4. 待办行按回车后，新行默认为普通文本行。
5. 空待办行回车时，取消 checkbox，变回普通空行。
6. 行首退格时，如果是待办行，先取消待办框；再次退格才与上一行合并。
7. 点击 checkbox 只切换完成状态，不破坏正文光标。
8. 便签列表页也支持直接点击待办 checkbox。

这个规则来自用户对 OPPO 便签的观察：待办作用于“当前逻辑行”，不是作用于 Markdown 标记。

### 5.3 文本格式

工具栏保留旧 UI 结构，提供：

- 正文
- H1 / H2 / H3 / H4 / H5
- 加粗
- 斜体
- 下划线
- 删除线
- 引用
- 代码
- 无序列表
- 有序列表

文本样式规则：

1. 有选中文字时，只处理选区。
2. 没有选中文字时，默认处理当前整行。
3. 如果目标范围已经全部有该样式，再点一次就是取消。
4. 如果目标范围只有部分有该样式，再点一次就是补齐整个范围。
5. 标题/正文不使用 toggle，而是明确设置当前行样式。

这个规则修复了早期“点击正文后反而变大加粗、点击标题后变小变细”的反向问题。

### 5.4 长按退格

用户测试发现：回车很多空行后，长按退格不会一直删除空白行。

修复规则：

- 空行里退格时，即使光标状态不是标准 `0,0`，也触发行合并/删除逻辑。
- 长按退格经过多行空白时，会持续向上删除空行。
- 待办空行仍按原规则：先取消待办框，再继续删除普通空行。

## 六、图片和附件

### 6.1 为什么作为特殊行

本轮明确：图片和附件不能插入文字中间。

原因：

- 如果图片/附件出现在正文字符流里，光标、退格、选中、复制粘贴都会变复杂。
- 这会复发早期待办行遇到的同类问题。
- 更稳定的方式是把图片和附件当成独立逻辑行/块。

最终规则：

```text
图片 = IMAGE 行
普通附件 = FILE 行
```

### 6.2 插入规则

1. 图片入口使用和主页底栏一致的系统图片选择方式：`GetContent("image/*")`。
2. 普通附件入口使用系统文件选择器：`OpenDocument("*/*")`。
3. 文件选中后通过 `NoteAttachmentStore.copyFromUri()` 复制到 App 私有目录。
4. `NoteDocument` 中只保存相对路径、附件名和 MIME。
5. 图片/附件插入到当前光标所在行之后。
6. 插入后，光标自动跳到附件/图片后的普通空行。

### 6.3 图片行

图片行直接渲染图片预览。

实现要点：

- 根据图片原始宽高比计算显示高度。
- 显示宽度贴合内容宽度。
- 设置最小高度和最大高度，避免过扁或超长图片破坏页面。
- 不再写死高度。

### 6.4 文件行

非图片附件统一使用通用文件卡片：

- 左侧是 Google Material 风格 file icon。
- 右侧显示附件名称。
- 外层为圆角矩形卡片。

本轮新增 `ic_note_file_filled.xml`，避免之前用字符方块导致图标显示成点状。

后续如果要适配 APK、PDF、音频、压缩包等，可以再从通用文件卡片中拆出专属样式。

### 6.5 选中和删除

图片/附件行支持选中状态。

删除规则：

1. 从下一行行首退格，先选中上一张图片/附件。
2. 再次退格删除该图片/附件行。
3. 删除时同步删除 App 私有目录中的文件。
4. 删除后如果没有普通文本行，自动补一个普通空行。

选中反馈经历过一次修复：

- 早期尝试修改图片背景，但图片会盖住背景，看不出选中状态。
- 后改为在图片/附件上方叠加半透明蒙层。
- 又发现蒙层可能错位，最终固定为内部 `FrameLayout`：底层内容，上层 `MATCH_PARENT` 蒙层。
- 这样蒙层天然贴合图片或附件卡片，不再上下错位。

## 七、便签导入导出和旧数据迁移

### 7.1 单个便签导入导出

导入导出入口放在便签编辑页底部工具栏 `+` 里。

原因：用户更多时候导出的是单个便签，不是全量备份。

导出规则：

- 无图片/附件：导出 JSON。
- 有图片/附件：导出 ZIP。

ZIP 结构：

```text
note.json
attachments/xxx.jpg
attachments/xxx.apk
```

导入规则：

- 支持 JSON / ZIP。
- 导入为一条新便签，不覆盖当前便签。
- ZIP 导入时，将附件复制回 App 私有目录，并重写 `attachmentPath`。
- 导入成功后自动打开新便签。

本轮修过一个导出 bug：

- 问题：有附件时仍然走 JSON 导出入口。
- 根因：UI 入口始终使用 `CreateDocument("application/json")`，没有根据附件存在与否切换 MIME。
- 修复：无附件使用 JSON launcher，有附件使用 ZIP launcher，并生成 `.zip` 文件名。

### 7.2 AI 分析

AI 分析保留在便签工具栏中。

规则：

- 只发送标题和文本/待办行内容。
- 跳过 `IMAGE` 和 `FILE` 行。
- 不发送附件路径、附件名称、MIME。

这样避免图片路径或本地文件信息被误传给 AI。

### 7.3 旧版便签自动迁移

旧版便签迁移由 `LegacyNoteMigrationCenter` 负责。

规则：

1. App 启动时自动迁移一次。
2. 扫描 `events` 表中 `tag = note / 便签` 的旧便签。
3. 标题使用旧 `Event.title`。
4. 正文使用旧 `Event.description`。
5. 只做纯文本迁移，不解析 Markdown。
6. 不删除旧 Event 数据。
7. 使用 `legacyMigrationKey` 避免重复迁移。

纯文本迁移示例：

```text
# 标题        -> 新版中显示为 # 标题
- [ ] 待办   -> 新版中显示为 - [ ] 待办
**加粗**     -> 新版中显示为 **加粗**
```

### 7.4 开发者选项

开发者选项新增：

- 手动迁移旧版便签。
- 清理旧版便签 Event 数据。

清理按钮只删除旧 `events` 表中的 note tag 数据，不影响新 `notes` 表。

## 八、悬浮窗便签接入

新版便签最终重新接入悬浮窗。

### 8.1 恢复能力

悬浮窗中恢复：

1. 底部输入模式可切换日程/便签。
2. 便签模式下输入内容创建新版 `notes` 便签。
3. 便签卡片显示标题、日期、分割线、正文/待办预览。
4. 右上角显示待办进度：如 `2/5` 或 `已完成`。
5. 待办可直接点击勾选。

7. 便签 icon 与底栏统一使用 `Icons.Outlined.StickyNote2`。
8. 便签颜色改为和日程一致的主题色，不再使用橙色。

### 8.2 滑动与编辑

参考旧版悬浮窗：

- 左滑：露出编辑按钮。
- 右滑：删除便签。
- 编辑按钮进入悬浮窗内联编辑。

内联编辑范围：

- 可编辑标题。
- 可编辑纯文本正文。
- 不编辑图片、附件、复杂格式。

这样避免在悬浮窗中再次引入复杂编辑器问题。

### 8.3 删除提示

悬浮窗删除提示经历过多次尝试：

1. 最初做了便签专属 Compose 浮层。
2. 后尝试复用日程归档浮层。
3. 又发现状态源和样式仍不一致。
4. 最终明确用户想要的是系统 Toast 样式：左侧 App icon，右侧文字。

最终规则：

- 日程归档：系统 Toast `已归档`。
- 便签删除：系统 Toast `已删除`。
- 不再使用 Compose 自绘 toast/undo 浮层。

## 九、日程重复展示问题

### 9.1 现象

用户反馈“全部日程”中出现同一日程重复展示，例如快递、体检、车辆相关事件。

### 9.2 排查过程

最初尝试直接读取设备数据库：

1. 使用 `adb devices` 确认设备连接。
2. 尝试 `run-as com.antgskds.calendarassistant` 读取数据库。
3. 由于安装包不是 debuggable，无法直接读取 `/data/data/.../events.db`。
4. 曾尝试 `adb backup`，但需要手机确认，随后停止，避免误操作。
5. 查询系统 Calendar Provider，发现系统日历里对应事件大多不是重复的。

随后用户手动导出备份：

```text
C:\Users\ASUS\Desktop\willdo_backup_20260601_005256.zip
```

只读分析结果：

```text
eventsJson.events 共 511 条
按 title + startTS + endTS + location + description 比对：
222 组重复，涉及 446 条记录
```

典型重复模式：

```text
本地行：source 为空，importId 为空
回流行：source = Caldav-x，importId = Caldav-x-xxx
```

截图中的 `中通快递 J-1-9148`、`体检`、`鄂J62017` 均符合该模式。

结论：不是 Compose 列表重复渲染，而是 Room 中已经存在不同 id 的重复业务行。

### 9.3 根因

系统日历同步回流时，主要按 `importId` 匹配已有 CalDAV 事件。

如果流程是：

```text
App 创建本地事件
-> 推送到系统日历
-> 系统日历回流
```

而本地原始事件没有及时绑定 `source/importId`，回流时就无法按 `importId` 找到它，最终插入第二条内容相同的新 Event。

### 9.4 修复

新增：

- `EventDuplicateSignature`
- `DuplicateEventCleanupCenter`

修复策略：

1. **升级后自动清理一次**
   - 使用 SharedPreferences 记录清理版本。
   - 避免每次启动都重复自动清理。

2. **开发者选项手动清理**
   - 新增“清理重复日程”按钮。
   - Toast 显示扫描数、删除数、合并数。

3. **合并策略**
   - 按内容签名分组。
   - 合并系统日历绑定、提醒、颜色、描述、附件引用。
   - 删除重复行。
   - `EventAttachmentsDao.moveAttachmentsToEvent()` 用于转移附件。

4. **防止再次产生重复**
   - `SystemCalendarSyncManager.fetchCalDAVCalendarEvents()` 中，如果按 `importId` 找不到 base，再按内容签名匹配未绑定本地事件。
   - 匹配到则绑定原记录，而不是插入新记录。

## 十、归档后从归档页消失

### 10.1 现象

ColorOS 设备上，日程归档后有时会从归档页消失；MIUI 设备不明显。

### 10.2 根因

`SystemCalendarSyncManager.fetchCalDAVCalendarEvents()` 从系统日历回流事件时，构造新的 `Event(...)` 没有保留 `base.archivedAt`。

触发流程：

```text
用户归档事件 -> archivedAt != null
ColorOS Calendar Provider 更频繁触发回流
回流构造 Event 时 archivedAt 丢失
insertOrUpdate 覆盖本地记录
archivedAt 变回 null
归档页中该事件消失
```

ColorOS 更容易出现，是因为它的 Calendar Provider / 同步服务更频繁触发回流；MIUI 触发少，所以不明显。

### 10.3 修复

本轮只修同步链路，不改 `DuplicateEventCleanupCenter`。

修复点：

1. 普通事件回流保留 `base?.archivedAt`。
2. 重复例外事件回流也保留 `base?.archivedAt`。
3. 本地未绑定事件被系统日历匹配绑定时，保留 `matchedUnbound.archivedAt`。
4. `isSameForSync()` 比较时忽略 `archivedAt`。
5. 系统日历缺失清理时，不删除已归档的本地绑定事件。

## 十一、Undo 与悬浮窗状态修复

### 11.1 现象

用户反馈：

- 已完成/已检票日程在悬浮窗左滑后仍显示 Undo。
- 手动修改 `endTS` 到当前或未来后，日程看起来应该重新进行中，但右上角仍显示已完成。
- 点击 Undo 后仍像已完成。
- 出现两个不同颜色的 Undo：一个主题色，一个黄色。

### 11.2 根因

1. 用户手动编辑时间只修改 `endTS`，没有重置 `state`。
2. 数据库中事件仍是 `STATE_COMPLETED` 或 `STATE_CHECKED_IN`。
3. 悬浮窗右上角优先看 state，因此仍显示“已完成/已检票”。
4. 当前代码把两类动作都用 Undo 表示：
   - 真正的 pending undo。
   - 已完成/已检票终态的“恢复/重开”。
5. 点击“恢复”又会创建新的 pending undo，形成 Undo 套 Undo。

### 11.3 参考旧版行为

查 `V1.2.6Beta`：

- 完成时保存原结束时间。
- `endDate/endTime` 改为当前时间。
- `completedAt = now`。
- 1 分钟内 Undo 可恢复原结束时间。
- 超过 1 分钟后 Undo 不应继续作为主要操作出现。

### 11.4 最终 UX

按用户确认后的规则实现：

1. 刚完成/检票后 1 分钟内：左滑显示黄色 Undo，编辑按钮仍为主题色。
2. 超过 1 分钟：终态日程左滑只显示编辑按钮。
3. 手动编辑终态日程时，如果保存后的 `endTS >= 当前时间`，自动把 `state` 改回 `STATE_PENDING`。
4. 重新激活后可再次完成/检票，并产生新的 1 分钟黄色 Undo。
5. 不再允许“已完成/已检票 -> 恢复 pending -> 再生成一个撤销恢复 Undo”的套娃行为。

### 11.5 修复点

`FloatingScheduleScreen.kt`：

- 状态按钮只在 pending undo 或未过期未完成时显示。
- pending undo 固定黄色 `0xFFFFA726`。
- 编辑按钮仍使用主题色。

`ScheduleCenter.kt`：

- 已完成/已检票但不在 pending undo 窗口时，状态动作直接忽略。
- `applyPatchToEvent()` 中，如果原状态是 `STATE_COMPLETED` 或 `STATE_CHECKED_IN` 且新 `endTS >= now`，自动恢复 `STATE_PENDING`。

## 十五、阶段结论

本轮最重要的结论是：便签不能继续作为日程 Event 的附属能力，也不应该用 Markdown 作为移动端主编辑模型。

最终方向：

```text
独立 notes 表
+ 结构化 NoteDocument
+ 原生行编辑器
+ 待办/图片/附件作为特殊行
+ Markdown 只作为未来导入导出边界
```

这套方案放弃了“完整 Markdown 编辑器”的想象空间，但换来了更稳定的光标、回车、删除、附件和悬浮窗待办体验，也更符合 Will do 作为智能日程助手而非专业 Markdown 编辑器的产品定位。

日程侧的核心结论是：Room 仍然必须作为唯一业务事实源，系统 Calendar Provider 只能作为互通层。同步回流必须尊重本地业务字段，例如 `archivedAt`、`state`、附件绑定和本地未绑定事件，否则 ColorOS 等 ROM 上会因为更频繁的 Provider 回流放大问题。
