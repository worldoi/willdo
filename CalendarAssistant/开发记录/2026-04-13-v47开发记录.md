## 二、重复日程操作完善

### 背景

原 APP 对重复日程仅支持两种编辑操作（仅修改此事件、修改所有事件），且删除操作没有重复专属选项，直接删除整个系列。用户希望对齐标准日历应用的功能，同时保持每个对话框只有 2 个按钮的简洁设计。

### 修改后的操作矩阵（2+2）

| 操作 | 类型 | 状态 |
|------|------|------|
| 仅修改此事件 | 编辑 | 已有（保持不变） |
| 修改此事件及以后 | 编辑 | 新增（替换"修改所有事件"） |
| 仅删除此事件 | 删除 | 新增 |
| 删除整个系列 | 删除 | 新增 |

### 实现细节

#### AppRepository.kt — 新增 3 个方法

**`modifyThisAndFutureInstances(parentEventId, sourceInstanceKey, modifiedEvent)`**

以选中实例的 `startDate` 为分界点：
- 过去的实例：直接转为独立事件（清除 recurring 标记，值不变）
- 更新 parent 事件（未来实例自动继承新值）
- 调用 `updateEvents` + `triggerAutoSync`

**`deleteSingleInstance(parentEventId, sourceInstanceId, sourceInstanceKey)`**

- 将 `instanceKey` 添加到 parent 的 `excludedRecurringInstances`
- 写入 Room `event_excluded_dates` 表（供 `syncRecurringMasters` 构建 EXDATE）
- 从 `currentList` 移除该实例
- 调用 `updateEvents` + `triggerAutoSync`

**`deleteEntireSeries(parentEventId)`**

- 异常事件（`isException && originalMasterId == seriesKey`）转为独立事件（清除 exception 标记）
- 从 `currentList` 移除 parent 和所有子实例
- Room 清理：`roomShadowWriter.deleteEvents`
- 调用 `updateEvents` + `triggerAutoSync`

**`modifyAllInstances`** — 保留方法作为内部工具，UI 不再暴露。

#### MainViewModel.kt — 新增 3 个包装方法 + 删除路由

- `modifyThisAndFutureInstances`、`deleteSingleInstance`、`deleteEntireSeries`
- `deleteEvent(event)` 方法增加重复事件路由：重复事件不直接删除，而是通过 `_recurringDeleteSession` StateFlow 触发 HomeScreen 中的删除确认对话框
- 新增 `recurringDeleteSession: StateFlow<MyEvent?>` 和 `dismissRecurringDeleteDialog()`

#### HomeScreen.kt — UI 修改

**编辑对话框**（修改原有对话框）：
```

正文: 此日程为重复事件，请选择修改范围：
按钮: [仅修改此事件] [修改此事件及以后] [取消]
```

**删除对话框**（新增）：
```

正文: 此日程为重复事件，请选择删除范围：
按钮: [仅删除此事件] [删除整个系列] [取消]
```

- 新增 `showRecurringDeleteDialog`、`pendingDeleteRecurringSession` 状态变量
- `LaunchedEffect` 观察 `recurringDeleteSession`，自动查找 parent 事件并创建 `RecurringEditSession`

#### SwipeableEventItem.kt — 滑动操作

## 三、重复日程反向同步 Twins 问题修复

### 背景

APP 内修改重复日程的单个实例后同步到系统日历，系统日历中同时显示修改后的单例和原始的重复实例（Twins）。

### 根因

1. 异常事件同步到系统日历时，未在父事件上添加 EXDATE，导致系统日历仍在该日期生成重复实例
2. MIUI Calendar Provider 不会自动添加 EXDATE（与 AOSP 行为不同）

### 修复措施

#### CalendarManager.kt

- **`getEventSyncId(eventId)`** — 查询系统日历事件的 `_sync_id` 字段，用于 Google 日历异常事件的创建
- **`appendExDate(eventId, newExcludedTimes)`** — 手动向系统日历父事件追加 EXDATE，解决 MIUI 不自动添加的问题
- **MANAGED_EVENT_MARKER** — 在 `buildRecurringEventContentValues` 中追加托管标记到 description，防止反向同步重新导入 APP 创建的重复事件

#### RoomEventShadowWriter.kt

- `toSeriesMaster` 新增 `existingMaster` 参数，保护 `system_recurring` 来源的 master 不被 INSERT OR REPLACE 覆盖

#### EventMasterDao.kt

- `getRecurringMasters` 查询条件加固：增加 `syncId IS NULL AND skipCalendarSync = 0 AND source != 'system_recurring'`

#### CalendarSyncManagerV2.kt

- `syncRecurringMasters`：启动时清理 `system_recurring` 来源 master 的孤儿 sync_map 条目
- `syncEvents`：异常事件处理分支使用 `createExceptionEvent` + `appendExDate` 在父日历中创建异常事件

## 四、通知栏完成重复实例持久化修复

### 背景

通过通知栏"已完成"按钮完成重复日程实例后：
1. APP 中立即呈现完成态（已过期）
2. 系统日历不呈现完成态，依旧保持重复态
3. 过一会，反向同步将 APP 内的完成态覆盖，重复态日程重新出现，通知栏重新弹出通知

### 根因

原 `completeScheduleEvent` 直接在重复子实例上设置 `isCompleted = true`。由于子实例仍保留 `isRecurring = true` 和 `parentRecurringId`，反向同步会用系统日历中未修改的重复实例覆盖完成状态。

### 修复方案：完成 = 异常 + EXDATE

对重复子实例的完成操作采用与 `modifySingleInstance` 相同的异常事件机制：

1. 将 `instanceKey` 添加到 parent 的 `excludedRecurringInstances`
2. 写入 Room `event_excluded_dates` 表（供同步时构建 EXDATE）
3. 移除旧的重复子实例
4. 创建新的异常事件（`isException = true`，标记 `isCompleted = true`，清除 recurring 关联，分配新 UUID）
5. 调用 `updateEvents` + `triggerAutoSync`

由于异常事件不再关联重复系列，反向同步不会覆盖其完成状态。EXDATE 确保系统日历不再在该日期生成重复实例。

### 修改

**AppRepository.kt** — `completeScheduleEvent` 方法重构：
- 方法包装在 `eventMutex.withLock` 中保证线程安全
- 新增重复子实例分支（异常事件 + EXDATE 流程）
- 找不到 parent 时 fallback 为普通事件处理
- 普通事件/课程保持原有逻辑不变
