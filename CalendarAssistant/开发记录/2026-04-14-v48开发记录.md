## 二、短信取件码链路重构

### 背景

用户反馈短信取件码自动识别不稳定。结合现有代码和参考项目排查后，发现当前问题不主要在正则，而主要在"短信怎么进来"这一步：

1. `SMS_RECEIVED` 在部分 ROM（尤其 MIUI/HyperOS）上不稳定
2. `SmsContentObserver` 只在 APP 进程存活后才有作用
3. `SmsNotificationListenerService` 已经写好，但之前没有真正接入 `AndroidManifest.xml`

参考 `parcel` 项目后，决定不照搬它的 UI/数据模型，而是吸收它最稳定的采集思路：

- **广播/通知负责触发**
- **短信库负责主取数**
- **通知正文只在查不到短信库时兜底解析**

### 最终架构

```
入口 1：SMS_RECEIVED
入口 2：SmsContentObserver
入口 3：SmsNotificationListenerService
        ↓
SmsImportCoordinator
        ↓
优先查询 content://sms/inbox
        ↓
SmsAnalysis.parse()
        ↓
SmsEventMapper -> MyEvent
        ↓
AppRepository.addEvent()
```

### 新增统一协调器

#### `core/sms/SmsImportCoordinator.kt`

新增短信导入协调器，作为三条入口的统一收口点。

职责如下：

1. **统一入口**
   - `importFromBroadcast()`
   - `importFromObserver()`
   - `importFromNotification()`

2. **统一时序**
   - 广播触发后延迟约 `1.2s`
   - 通知触发后延迟约 `1.5s`
   - 等待系统短信库完成写入，再查询 `content://sms/inbox`

3. **统一主取数策略**
   - 广播通道：按发件人 hint 查询最近短信
   - 通知通道：按通知正文 hint 查询最近短信
   - Observer 通道：直接使用 `_id > lastProcessedId` 查出的新短信行

4. **统一兜底策略**
   - 若短信库查询不到匹配记录，则回退解析广播正文/通知正文

5. **统一线程安全与去重**
   - 使用 `Mutex` 保证多通道并发时不会重复入库
   - 入库前统一走 `SmsEventMapper.isDuplicate()`

### 新增统一事件映射器

#### `core/sms/SmsEventMapper.kt`

新增短信事件映射器，统一负责：

- `CalendarEventData -> MyEvent` 转换
- 同类短信事件去重（`tag + description + 未过期`）
- 与识别默认时长保持一致，统一使用识别时长模式计算结束时间

这样可以彻底删除各通道中重复的 `eventDataToMyEvent()` 和本地去重逻辑。

## 三、三通道重构细节

### 1. `SmsReceiver.kt` — 从"直接解析入库"改为"广播触发器"

旧逻辑：

- 直接解析 PDU
- 直接调用 `SmsAnalysis.parse()`
- 直接转换并入库

新逻辑：

- 仍接收 `SMS_RECEIVED`
- 仅提取发件人 hint 和广播正文 fallback
- 调用 `SmsImportCoordinator.importFromBroadcast()`

这样做的好处是：

- 不再依赖广播正文就是最终真相
- 优先走短信库，提高稳定性
- 保留非 MIUI ROM 下的标准广播触发能力

### 2. `SmsContentObserver.kt` — 从"自己解析"改为"统一转发"

旧逻辑：

- 查询新短信后直接解析并入库

新逻辑：

- 继续维护 `lastProcessedId`
- 继续监听 `content://sms`
- 查询到新短信后，封装为 `SmsInboxRow`
- 统一交给 `SmsImportCoordinator.importFromObserver()`

保留的能力：

- 500ms 防抖
- 进程存活时的数据库补漏能力
- `READ_SMS` 权限不足时的安全失败

### 3. `SmsNotificationListenerService.kt` — 正式改造成主触发通道

本次将其从"备用代码"改成真正可用的主触发通道。

新逻辑：

1. 只监听系统短信应用通知包名
   - `com.android.mms`
   - `com.miui.mms`
   - `com.google.android.apps.messaging`
   - 以及华为/三星/OPPO/一加等常见短信包名

2. 收到通知后先提取正文

3. 同一内容 2 秒内去重

4. 调用 `SmsImportCoordinator.importFromNotification()`

5. 在协调器内部：
   - 优先查短信库
   - 查不到再回退解析通知正文

这次没有接入第三方 App 通知（微信、拼多多、抖音等），首版范围仅限**系统短信 App 通知**，避免误识别范围过大。

## 十、补充记录（侧边栏唤起悬浮窗修复）

### 背景

用户反馈两个连续问题：

1. 开启侧边栏后，从侧边栏区域横滑无法呼出悬浮窗，反而经常触发系统返回手势
2. 长按音量+仍可正常呼出悬浮窗，说明悬浮窗本体与权限链路正常，问题集中在 `EdgeBarService` 唤起链路
3. 在悬浮窗中触发一次识屏后，侧边栏后续无法再次呼出悬浮窗

### 根因分析

#### 1. 侧边栏缺少稳定的边缘手势占用

- 侧边栏本质上是第三方 Overlay，可见宽度虽然可调，但如果没有把整块贴边区域明确声明为手势排除区，系统返回手势仍可能优先吃掉横滑
- 对用户来说，表现就是：侧边栏明明看得见、也能调宽，但横滑时仍被系统当成返回操作

#### 2. 识屏后悬浮窗只隐藏未真正关闭

- 识屏流程原来只调用 `hideFloatingWindow()` 隐藏窗口
- `FloatingScheduleService.isShowing` 没有同步恢复为 `false`
- `EdgeBarService` 触发条件又要求 `!FloatingScheduleService.isShowing`
- 于是会进入"窗口已经隐藏，但逻辑上仍被判定为显示中"的假存活状态，导致侧边栏后续无法再次拉起悬浮窗

### 修复方案

#### `service/floating/EdgeBarService.kt`

- 保留现有自定义侧边栏方案，不取消用户可配置的宽度/高度/透明度能力
- 将侧边栏重新贴边显示（`x = 0`），让用户设置的**侧边栏宽度**直接作为边缘触摸热区宽度
- 在 Android 10+ 上为整块侧边栏区域设置 `systemGestureExclusionRects`
- 继续保留：
  - 横向滑动触发悬浮窗
  - 纵向拖动调整位置
- 为 `startService(FloatingScheduleService)` 增加失败兜底，避免启动失败后侧边栏被错误隐藏

#### `service/floating/FloatingScheduleService.kt`

- 保持"识屏后不自动恢复悬浮窗"的产品行为不变
- 但实现从"仅隐藏窗口"改为：
  1. 启动无障碍识屏
  2. 完成回调
  3. 调用 `requestClose()` 真正关闭 `FloatingScheduleService`
- 同时为普通启动路径补上 `showFloatingWindow()`，避免服务实例仍在但窗口已隐藏时无法重新显示

### 最终效果

1. 侧边栏宽度越大，覆盖系统返回手势区域的范围越大，用户可以通过调宽提升命中率
2. 侧边栏横滑恢复为稳定呼出悬浮窗，而不是频繁触发返回
3. 悬浮窗执行一次识屏后，不会再卡在"逻辑显示中 / 实际已隐藏"状态
4. 侧边栏和长按音量+在识屏后都可以再次正常呼出悬浮窗

## 十三、补充记录（重复日程浮层定位修复）

### 背景

用户反馈重复事件的"编辑范围"与"删除范围"浮层没有像"正在识别"浮层那样贴底显示，而是直接跑到了屏幕顶部。

### 根因

排查后发现问题不在 `FloatingActionCard` 本身，而在挂载位置：

1. `FloatingActionCard` 自身不会自动固定到底部，必须由外层 `BoxScope` 显式 `align(Alignment.BottomCenter)`
2. 重复事件的编辑/删除浮层被写到了 `HomeScreen` 根 `Box` 之外，已经失去 `BoxScope`
3. 由于只剩 `padding(bottom = ...)`，Compose 最终按默认布局把卡片放到了顶部

### 修复方案

#### `ui/page_display/HomeScreen.kt`

- 将重复事件编辑浮层和删除浮层整体移回 `HomeScreen` 根 `Box` 的覆盖层
- 根容器明确改为 `Modifier.fillMaxSize()`，确保遮罩与浮卡的覆盖范围稳定
- 两个浮层统一改为：
  - 遮罩层：全屏半透明黑色背景
  - 卡片层：`Modifier.align(Alignment.BottomCenter).padding(bottom = floatingBarOffset + 16.dp)`
- 视觉与交互对齐现有"正在识别"浮层

### 最终效果

1. 重复事件编辑浮层恢复贴底显示
2. 重复事件删除浮层恢复贴底显示
3. 与底部悬浮条、识别浮层的视觉层级保持一致

## 十四、补充记录（侧边栏顶部间距恢复）

### 背景

此前为了去掉侧边栏上下 padding，只保留了 `statusBarsPadding()`。结果虽然没有顶到状态栏内容，但视觉上已经直接贴在通知栏下沿，缺少原有的呼吸感。

### 历史值确认

回看旧实现后确认：原侧边栏使用的是四边统一 `16.dp` padding，也就是说顶部原本额外保留了 `16dp` 的视觉间距。

### 修复方案

#### `ui/components/SettingsSidebar.kt`

- 保持当前左右 `16.dp` 不变
- 保持 `statusBarsPadding()`，继续先避开系统状态栏
- 在其后补回 `.padding(top = 16.dp)`，恢复与历史版本一致的顶部间距

### 最终效果

1. 侧边栏不会再视觉上紧贴通知栏下沿
2. 主页面与设置详情页共用侧边栏都同步恢复到更自然的顶部留白

## 十五、补充记录（识别默认持续时长统一设置重构）

### 背景

用户希望可自定义自动识别事件的默认持续时长。排查现状后发现：

1. AI 提取侧主要依赖提示词中的默认结束时间规则
2. 短信取件码链路则在代码中写死为 `1h`
3. 同一产品行为被分散在 prompt、AI fallback、短信映射三处，后续维护和解释成本都很高

由于当前版本尚未正式发布，仅运行在开发设备上，因此本次决定直接废弃临时的分钟滑杆方案，收口为离散模式设置，不再背负兼容包袱。

### 新设计

统一将默认持续时长改为 5 档：

- `1H`
- `2H`（默认）
- `3H`
- `6H`
- `当天结束`

统一规则如下：

1. 若原文明确给出结束时间 / 持续时长 / 截止时间，优先使用原文
2. 若原文未给出明确结束时间：
   - 小时档位：`endTime = startTime + xH`
   - 当天结束：`endTime = 事件开始日期 23:59`
3. AI prompt、AI 落地 fallback、短信取件码导入三条链路统一使用同一规则

### 实现细节

#### `data/model/RecognitionDurationDefaults.kt`

- 新增 `RecognitionDurationMode` 枚举：`ONE_HOUR`、`TWO_HOURS`、`THREE_HOURS`、`SIX_HOURS`、`END_OF_DAY`
- 提供：
  - `displayLabel()`
  - `promptRuleText()`
  - `resolveRecognitionEnd(start)`

#### `data/model/MySettings.kt`

- 用 `recognitionDurationMode` 替换之前的分钟值字段
- 默认值设为 `RecognitionDurationMode.TWO_HOURS`

#### `ui/viewmodel/SettingsViewModel.kt`

- `updatePreference()` 改为写入 `recognitionDurationMode`

#### `ui/page_display/settings/PreferenceSettingsPage.kt`

- 删除分钟滑杆入口
- 改为 `ActionSettingItem` + `AlertDialog` 单选弹窗
- 用户可直接选择 `1H / 2H / 3H / 6H / 当天结束`

#### `core/ai/AiPrompts.kt`

- Prompt 规则补丁改为动态输出：
  - `endTime = startTime + xH`
  - 或 `endTime = 当天 23:59`
- 提示词缓存版本从 `v7` 提升到 `v8`，强制刷新旧缓存

#### `core/ai/RecognitionProcessor.kt`

- AI 文本、日程、取件、多模态四条 prompt 生成路径统一读取 `RecognitionDurationMode`

#### AI / 短信落地链路

- `core/ai/AiEventDraftConverter.kt`
- `service/accessibility/TextAccessibilityService.kt`
- `service/floating/FloatingScheduleService.kt`
- `core/sms/SmsAnalysis.kt`
- `core/sms/SmsEventMapper.kt`
- `core/sms/SmsImportCoordinator.kt`

上述链路全部统一走 `resolveRecognitionEnd(...)` 或同一模式参数，不再各自写死 `+1h`。

### 说明

本次是对同一版本内的临时实现做直接收口：

- 保留"软件设置统一控制"的大方向
- 但把最初的分钟制实现升级为更符合产品认知的离散档位
- 因为尚未正式发布，所以不额外做旧分钟字段迁移
