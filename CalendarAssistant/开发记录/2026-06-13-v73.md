## 二、每日提醒胶囊标题调整

### 2.1 问题背景

每日提醒接入实况胶囊后，原先沿用普通通知的标题文案：

```text
今日日程提醒 24°C 阴
明日日程预告 24°C 阴
```

在胶囊态下显得偏长，也和软件内其它识别结果通知的紧凑标题风格不一致。

现有通知中识别结果类标题已使用无空格竖线分隔，例如：

```text
识别成功|会议
识别日程|取件
```

因此本轮每日提醒也统一为同类风格。

### 2.2 新标题规则

每日提醒现在拆分为完整标题和短标题：

- 早上提醒完整标题：`今日提醒|24°C 阴`
- 晚上预告完整标题：`明日预告|24°C 阴`
- 无天气数据时：`今日提醒` / `明日预告`
- 胶囊短标题固定使用：`今日提醒` / `明日预告`

对应模型层新增 `DailySummaryPayload.shortTitle`，通知层不再根据 `isMorning` 自行拼短标题，而是直接使用 payload 提供的文案。

## 三、汇总内容改为展示项来源

### 3.1 原逻辑风险

旧版每日提醒直接从原始 `Event` 列表按 `startDate == targetDate` 过滤。

这会带来两个问题：

- 重复日程母事件没有经过展示层展开，今天/明天实际发生的重复实例可能不会出现在提醒中。
- 跨天日程如果开始日期不是目标日期，但仍覆盖目标日期，也可能被漏掉。

### 3.2 新逻辑

`LocalDailySummaryQueryApi` 现在改为复用：

```kotlin
ScheduleDisplayHelper.buildDisplayItems(events, targetDate, targetDate)
```

这样每日提醒拿到的是和 UI 展示一致的目标日期展示项：

- 普通日程按日期范围 overlap 判断。
- 重复日程母事件会展开为目标日期实例。
- 已有重复例外子事件作为真实单次展示项参与汇总。
- 已归档日程由 `ScheduleDisplayHelper` 统一过滤。

在此基础上，每日提醒继续保留原有业务过滤：

- 过滤已完成日程。
- 早上提醒过滤已经结束的日程。
- 按 `startTS` 和 `title` 排序。

## 四、完整/精简模板内容

每日提醒 payload 现在显式携带模板行内容：

- `fullLines`：目标日期所有日程标题，按开始时间排序。
- `compactLines`：只有一个日程时显示该标题；多个日程时显示最早的一个标题，以及 `以及其他 x 个日程`。

胶囊完整模板会列出所有目标日期日程标题，精简模板会收敛为两行概要。

普通通知本轮没有单独重设计，仍使用 `payload.title` 和 `payload.content`，后续如有需要再针对普通通知栏样式单独调整。

## 八、非日程类实况胶囊标题规范化

### 8.1 调整范围

本轮明确保持日程类通知现状不变，日程、每日提醒、取件、火车、航班、打车、课程、票务等事件类胶囊不纳入标题重构。

实际调整范围限定为非日程类：

- 天气预警 / 天气风险。
- OCR 分析中 / OCR 结果。
- 本地模型加载状态。

网络速度等无关状态保持原样。

### 8.2 标题规则

非日程类实况胶囊统一拆成两个标题层级：

- `shortText`：胶囊态短标题，只保留 `xx`。
- `primaryText`：通知展开态标题，尽量使用 `xx|yyyy`。

`primaryText` 继续受 11 个中文字符以内的约束。若明细为空、重复或无法在长度预算内自然表达，则回退为短标题。

示例：

```text
暴雨预警|红色暴雨
降雨风险|一小时后
正在分析|屏幕内容
模型加载中|本地模型
```

### 8.3 天气类处理

天气标题不再把位置拼进标题，位置仍保留在内容正文中。

官方天气预警标题按预警事件和等级压缩，例如：

```text
暴雨预警|红色暴雨
暴雨解除|红色暴雨
```

天气风险标题按风险类型和中文时间表达压缩：

- 30 分钟以内使用 `即将出现`。
- 约 1 小时使用 `一小时后`。
- 约 2 小时 / 3 小时等使用 `两小时后` / `三小时后`。
- 无可靠时间信息时只保留风险短标题。

### 8.4 OCR 与本地模型

OCR 分析链路来自无障碍截图分析流程，进度态使用 `正在分析` 作为短标题，并从标题或内容中提取 `屏幕内容` 等明细。

OCR 结果态和本地模型加载态同样优先保留短标题，并把可读明细放进 `primaryText` 的竖线后半段；常见长文案会做简化，例如：

- `未识别到有效日程` 简化为 `无有效日程`。
- `没有可入库的新事件` 简化为 `无新事件`。
- `正在分析屏幕内容` 简化为 `屏幕内容`。

## 九、小米超级岛标题映射

### 9.1 原行为

小米超级岛请求原先统一使用：

```kotlin
display.primaryText.ifBlank { display.shortText }
```

作为 `MiuiIslandRequest.title`。

在非日程类标题改为 `xx|yyyy` 后，小米超级岛主标题会直接显示完整展开态标题，例如：

```text
暴雨预警|红色暴雨
正在分析|屏幕内容
模型加载中|本地模型
```

这和“胶囊态标题显示 xx”的目标不一致。

### 9.2 新映射

`MiuiIslandManager` 现在只对非日程白名单使用短标题作为超级岛主标题：

- `TYPE_WEATHER_ALERT`
- `TYPE_OCR_PROGRESS`
- `TYPE_OCR_RESULT`
- `TYPE_MODEL_LOADING`

其它类型全部保持原逻辑，因此日程类、取件、出行、课程、票务等事件胶囊不受影响。

当非日程内容为空时，会尝试从 `primaryText` 的 `|` 后半段提取明细作为内容兜底，避免主标题和内容都显示同一个 `xx`。

示例变化：

```text
原：title = 暴雨预警|红色暴雨，content = 10:30发布 · 预警正文
新：title = 暴雨预警，content = 10:30发布 · 预警正文

原：title = 正在分析|屏幕内容，content = 正在分析屏幕内容...
新：title = 正在分析，content = 正在分析屏幕内容...
```

## 十、Flyme 实况通知确认

Flyme 实况通知本轮未做代码修改，只确认其字段映射和原生实况通知基本一致：

- 原生实况通知胶囊短文本使用 `display.shortText`。
- 原生通知标题和展开标题使用 `display.primaryText`。
- Flyme `notification.live.capsuleContent` 使用 `display.shortText`。
- Flyme `setContentTitle`、`BigTextStyle` 和 `android.title` 使用 `display.primaryText`。

因此前面建立的 `shortText = xx`、`primaryText = xx|yyyy` 可以自然同步到 Flyme：胶囊态显示短标题，通知/展开态显示完整标题。

Flyme 自定义 `RemoteViews` 的主文本目前也使用 `display.primaryText`，等同于通知卡片标题。如果后续希望 Flyme 通知栏小卡片收起态也只显示 `xx`，再单独拆分 compact/big 标题；本轮先保持与原生展开态一致。
