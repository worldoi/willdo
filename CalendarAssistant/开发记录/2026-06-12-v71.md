## 二、语音录入路线复盘与最终方向

### 2.1 讯飞 SDK 试验

本轮曾接入讯飞 MSC Android SDK 做独立验证，确认：

- SDK 包可接入工程并通过编译。
- `RecognizerDialog` 能作为前台测试入口启动。
- 真机报错 `20006` 最终确认与用户未授予麦克风权限有关，不是服务未开通。

后续用户确认本地录音链路在授予权限后也能跑通，因此本轮决定回收讯飞 SDK 试验，不作为主链路：

- 删除讯飞 jar/native/assets/demo 入口残留。
- 移除实验室中的讯飞测试入口。
- 保留本地 Sherpa/QuickMemoAudioRecorder 作为默认语音随口记路线。

### 2.2 权限与后台限制结论

本轮关键结论：

- 未授予 `RECORD_AUDIO` 会导致 SDK 或本地录音在底层 `AudioRecord/MediaRecorder` 阶段失败。
- Android/ColorOS 对 `FOREGROUND_SERVICE_TYPE_MICROPHONE` 的后台限制仍然存在，但悬浮窗内本地录音在权限正确时可工作。
- 后续所有语音入口必须先显式检查录音权限，不能直接启动录音器。

`FloatingCenter.startVoiceCaptureService()` 已增加麦克风权限检查，权限缺失时拉起 `MainActivity` 请求权限，不再直接进入录音流程。

## 三、音量+长按语音触发

### 3.1 设置项简化

音量+长按动作从四档调整为三档：

```text
识屏 | 悬浮窗 | 语音
```

原先的“无操作”不再作为滑块选项，旧数据通过 `coerceIn(1, 3)` 归一化。

### 3.2 触发规则

新的触发规则：

- 选择“语音”：长按音量+ 直接启动本地语音随口记录音。
- 选择“悬浮窗”：
  - 若悬浮窗未显示，先打开悬浮窗。
  - 若用户持续按住不松手，悬浮窗出现后自动开始录音。
  - 若悬浮窗已显示，再次长按音量+ 直接开始录音。
- 松开音量+ 时停止录音并进入保存确认流程。

### 3.3 短按音量键保护

短按音量+ 仍通过 `AudioManager.adjustSuggestedStreamVolume(...)` 模拟系统音量增加，避免长按过滤影响普通音量调节。

## 四、语音转写稳定性与 Proguard 修复

### 4.1 Sherpa JNI 保留规则

真机 release 包中出现 Sherpa JNI 崩溃：

```text
NoSuchFieldError: OfflineRecognizerConfig.decodingMethod
JNI DETECTED ERROR: fid == null
```

原因是 R8 混淆了 `com.k2fsa.sherpa.onnx` 下的 Java 配置字段，native 层按原字段名查找失败。

已在 `app/proguard-rules.pro` 增加：

```pro
-keep class com.k2fsa.sherpa.onnx.** { *; }
-dontwarn com.k2fsa.sherpa.onnx.**
```

### 4.2 转写状态收敛

语音随口记曾出现长期停留在“转写中”的问题。根因是进程崩溃、native 异常或转写超时后，数据库状态可能停在 `PROCESSING`。

本轮补齐：

- `QuickMemoDao.getUnfinishedVoiceMemos()` 查询 `PENDING/PROCESSING` 的语音随口记。
- `QuickMemoCenter.start()` 启动时恢复未完成转写。
- 单条转写增加超时控制，当前为 `120_000ms`。
- 失败、超时或音频路径缺失时统一落为 `FAILED`，避免 UI 永久显示“转写中”。
- 使用 `activeTranscriptionIds` 避免同一条语音随口记重复转写。

## 五、随口记排序与待办规则

### 5.1 sort_rank 持久化

为支持悬浮窗拖拽排序和随口记页统一顺序，`quick_memos` 表新增：

```text
sort_rank INTEGER NOT NULL DEFAULT 0
```

数据库从 version 8 升级到 version 9，迁移时按旧规则回填：

```text
ACTIVE 待办 -> 普通随口记 -> COMPLETED 待办
同组内按 updated_at DESC / id DESC
```

DAO 默认排序调整为：

```sql
ORDER BY sort_rank ASC, updated_at DESC
```

### 5.2 新建与激活待办的插入规则

规则调整为：

- 新建文本/语音随口记默认插入最上方。
- 普通随口记被标记为待办时插入最上方。
- 手动拖拽排序后不再强制代办置顶，尊重用户排序。
- 完成/撤回待办不自动改变排序。

### 5.3 悬浮窗拖拽排序

悬浮窗随口记列表支持长按拖拽排序：

- 长按卡片后震动反馈。
- 拖拽时卡片放大并提高层级。
- 其他卡片通过 `animateItem()` 实时让位。
- 当前拖拽卡片禁用 `animateItem()`，避免拖动交换时出现双重位移动画闪烁。
- 松手后通过 `QuickMemoCenter.updateSortRanks(ids)` 写入持久顺序。

## 六、随口记待办交互

### 6.1 长左滑行为

随口记卡片长左滑规则：

```text
普通随口记 -> 转为待办
待办随口记 -> 标记完成
已完成待办 -> 撤回完成
```

本轮不使用一分钟撤销态。

### 6.2 旧状态闭包问题修复

曾出现普通随口记长滑转为待办后，再次长滑无法标记完成的问题。

根因是横向手势使用：

```kotlin
pointerInput(memo.id)
```

手势协程只按 `memo.id` 重建，`todoState` 变化后仍捕获旧的 `isTodo=false`，第二次长滑继续调用“转为待办”。

已修复为：

```kotlin
pointerInput(memo.id, memo.todoState)
```

使待办状态变化后手势逻辑同步更新。

### 6.3 撤回完成图标区分

侧滑待办按钮中：

- `转为待办` / `完成待办` 使用 `CheckCircle`。
- `撤回待办` 使用 `Undo`。

避免完成和撤回完成使用同一个图标造成语义混淆。

## 七、日程候选通知与详情页

### 7.1 转写后自动分析

语音随口记转写成功后继续调用识别链路，将文本送入 AI 分析，生成 `QuickMemoSuggestionEntity` 日程候选。

候选通过 `QuickMemoSuggestionCodec` 保存 `RecognitionDraft` JSON。

### 7.2 识别日程通知

随口记 AI 识别出日程候选后发送通知：

```text
识别日程|标题
```

通知内容展示时间、地点或备注，并复用近期新增日程/分析失败通知的样式能力。

普通通知和实况通知均适配：

- 普通通知显示一个 `添加` 按钮。
- 实况通知通过 `CapsuleActionSpec` 携带 `suggestion_id`，支持同一个 `添加` 动作。
- 通知 60 秒后自动注销。
- 用户可直接划掉或忽略通知，相当于取消。

### 7.3 独立随口记详情页

新增 `quick_memo_detail/{memoId}` 路由。

详情页结构：

- 顶部返回与标题。
- 语音播放/转写状态区域。
- 转写文本或随口记正文编辑区域。
- 如果存在待处理日程候选，则展示推荐日程创建卡片。

列表点击和通知点击均可进入详情页。

## 八、悬浮窗随口记卡片 UX

### 8.1 卡片信息架构

悬浮窗随口记卡片统一为轻量结构：

```text
颜色标签 | 一行文本摘要 | 语音播放按钮（语音卡片）
```

取消折叠态中的日期、随口记标签、待办标签、已完成标签等冗余元素。

### 8.2 文本与语音卡片统一

文本随口记：

- 中间展示一行正文。
- 超出部分使用省略号。
- 蓝色标签表示普通随口记。
- 黄色标签表示待办随口记。

语音随口记：

- 中间先展示转写文本摘要。
- 下方保留轻量 Siri 风格波形。
- 右侧保留播放按钮。
- 默认折叠态也能看到语义内容，避免只有波形导致信息密度过低。

### 8.3 波形绘制优化

语音波形从简单线段绘制改为 Path 绘制：

- 多层曲线共享中心轴。
- 使用 `sin(progress * PI).pow(2)` 作为包络。
- 通过 `rememberInfiniteTransition` 驱动相位。
- 播放中提高振幅，静止时保留轻微呼吸感。

### 8.4 完成态视觉弱化

已完成待办随口记不改变卡片背景，只弱化：

- 左侧颜色标签。
- 正文颜色。
- 语音波形颜色。
- 播放按钮颜色。

整体风格对齐悬浮窗日程卡片的过期/完成态，而不是额外做重背景。

## 十一、软件更新入口

### 11.1 远程 JSON 数据源

本轮新增软件更新检查链路，复用现有 Gitee + Cloudflare Worker 托管方案，但不复用 Prompt 更新业务代码。

新增 `APP_UPDATE_URL` 构建字段，默认指向：

```text
https://prompt.aixinjueluoonline.top/update.json
```

远程 JSON 当前采用：

```json
{
  "versioncode": 68,
  "versions": []
}
```

其中：

- 顶层 `versioncode` 用于和 `BuildConfig.VERSION_CODE` 比较。
- `versions` 用于展示历史版本日志。
- 单个版本支持 `versionname`、`downloadUrl`、`downloadPassword` 和 `sections`。
- `sections` 下按 `title + items` 展示新增、修复、优化等内容。

### 11.2 检查器与状态

新增 `AppUpdateChecker`，使用 Ktor 拉取远程 JSON，并通过 Kotlin Serialization 解析为 `RemoteAppUpdateInfo`。

检查逻辑保持简单：

```text
remote.versioncode > BuildConfig.VERSION_CODE -> 有更新
否则 -> 当前已是最新
```

`MainViewModel` 新增 `AppUpdateUiState`，应用启动时静默检查，不弹主动更新弹窗。

### 11.3 侧边栏入口与红点

设置侧边栏顶部操作区将“退出应用”替换为“软件更新”。

当云端 `versioncode` 大于本地 `VERSION_CODE` 时：

- 侧边栏“软件更新”右侧显示红点。
- 不主动弹窗打断用户。
- 用户进入软件更新页后查看更新日志和下载入口。

### 11.4 软件更新页

新增 `AppUpdatePage`：

- 移除顶部“当前版本/云端 versionCode/手动检查”卡片，避免向普通用户暴露无意义字段。
- 只展示版本日志卡片。
- 最新版本默认展开，历史版本默认折叠。
- 点击版本卡片或右侧箭头可展开/收起。
- 如果 `downloadUrl` 非空，显示 `前往下载` 按钮并交给系统浏览器打开。
- 如果 `downloadPassword` 非空，在按钮上方显示 `提取码`。

蓝奏云等带提取码的下载页不做 WebView 自动填充，避免不稳定、易失效和潜在合规风险。

## 十二、每日日程提醒维护

### 12.1 过期日程误提醒修复

用户反馈早上收到“每日日程提醒”时，通知中仍包含已经结束的日程。

根因有两点：

- 原始早报只按 `startDate == today` 过滤，没有排除已结束日程。
- 原始调度使用 `setInexactRepeating`，系统可能将 06:00 早报延迟到更晚触发。

本轮调整 `LocalDailySummaryQueryApi`：

- 早报只包含今天未完成、且 `endTS > now` 的日程。
- 晚报仍展示明天全天日程。
- 所有汇总日程按开始时间和标题排序。
- 空标题统一显示为 `未命名日程`。

### 12.2 胶囊通知适配

每日提醒接入实况胶囊，但不新增独立设置，复用现有：

```text
isLiveCapsuleEnabled
liveNotificationTemplateMode
```

开启实况胶囊时，`NotificationCenter.showDailySummaryNotification(...)` 复用 `NotificationTemplateCenter.composeBody(...)`：

- 详细模板：展示全部日程标题。
- 精简模板：第一行显示第一个日程，第二行显示 `以及其他 x 个日程`。
- 自动模板：沿用系统能力判断，多行能力足够时走详细模板，否则走精简模板。

关闭实况胶囊时，仍发送原普通通知，保留 `BigTextStyle` 展示完整内容。

### 12.3 通知 ID 与自动消失

新增每日提醒专用通知 ID：

```text
DAILY_SUMMARY_MORNING
DAILY_SUMMARY_EVENING
```

避免与普通日程提醒、日程候选通知、识别结果通知互相覆盖。

每日提醒胶囊设置 60 秒自动取消，避免早报/晚报作为 ongoing 胶囊长期停留。

### 12.4 单次精确调度

`DailySummaryReceiver.schedule()` 从 `setInexactRepeating` 改为每日单次调度：

- 应用例行启动时安排下一次 06:00 和 22:00。
- 每次 Receiver 触发后，在 `finally` 中重新安排同类型下一次提醒。
- Android M 及以上优先使用 `setExactAndAllowWhileIdle`。
- 遇到精确闹钟权限或配额异常时降级为普通 `set`。

这样可以减少重复闹钟在系统延迟后继续发出过期内容的概率，同时保持无权限场景下仍能尽力提醒。
