## 二、便签置顶

### 问题

新版便签列表只能按更新时间排序，无法固定重要便签。后续如果便签数量增加，用户很难把长期使用的便签放在顶部。

### 方案

新增 `pinned_at` 字段，而不是简单 boolean。

原因：

- boolean 只能表示是否置顶，不能表达置顶时间。
- `pinned_at` 可以用于排序，后置顶的便签排在更前面。
- 未来如果做手动排序，也可以继续兼容。

实现点：

- `NoteEntity` 新增 `pinnedAt: Long?`。
- Room 数据库从 version 6 升级到 version 7。
- 新增 `MIGRATION_6_7`：

```sql
ALTER TABLE notes ADD COLUMN pinned_at INTEGER DEFAULT NULL;
CREATE INDEX IF NOT EXISTS index_notes_pinned_at ON notes(pinned_at);
```

- `NotesDao.observeNotes()` 排序改为：

```text
置顶在前 -> pinned_at DESC -> updated_at DESC
```

- `NoteRepository.saveNote()` 保存正文时保留已有 `pinnedAt`，避免编辑便签后丢失置顶状态。
- `NoteCenter` / `MainViewModel` 增加 `setPinned()`。

### UI 迭代

最初在便签卡片标题下方显示 `pin icon + 已置顶`。真机查看后发现左侧信息过于拥挤，视觉不协调。

最终调整为：

- 置顶 icon 放到标题右侧。
- 只保留一个 `PushPin` icon，不再显示“已置顶”文字。
- icon 放大到 `18.dp`。
- 颜色使用主题色并降低透明度。

### 放弃的入口

曾尝试在便签页长按弹出的 `FloatingActionCard` 中加入“置顶/取消置顶”按钮。测试后发现这破坏了原本“长按删除确认”的简单语义，因此恢复为原来的删除确认样式。

当前保留的置顶入口：编辑页右上角 pin 按钮。

## 三、搜索范围扩大

### 问题

旧搜索只查 `title/plainText`，但新版便签已经有结构化内容：

- 待办行。
- 图片块。
- 文件附件块。
- 附件名和 MIME。

如果用户记得的是附件文件名或图片文件名，旧搜索无法命中。

### 方案

新增 `NoteDocument.searchableText(title)`。

搜索范围包括：

- 标题。
- 所有文本行。
- 待办文本。
- 图片/附件名称。
- MIME。
- 附件路径最后一级文件名。

第一版继续在内存中过滤。原因是当前便签数量不会太大，引入 FTS 会增加迁移和维护成本。后续如果便签量明显增加，再考虑单独做 FTS 表。

## 四、悬浮窗只编辑普通文本，保护复杂结构

### 问题

悬浮窗内联编辑便签时，原来会使用：

```kotlin
NoteDocument.fromPlainText(body)
```

这会把整篇便签重建为纯文本文档。如果便签里有待办、图片、附件、标题样式或文本 span，悬浮窗保存后会丢失结构。

### 方案

明确悬浮窗只负责普通文本：

- 悬浮窗只显示普通文本行。
- 不显示待办、图片、附件、分割线和格式行。
- 悬浮窗保存时只更新普通文本行。
- 原文档中的待办、图片、附件、格式内容继续保留。

新增方法：

- `NoteDocument.isFloatingPlainTextLine()`。
- `NoteDocument.floatingText()`。
- `NoteDocument.withFloatingText(text)`。

最终行为：复杂便签可以在悬浮窗里快速改普通正文，但不会覆盖特殊块。

## 五、快捷命令：/t /d /i /f

### 需求

用户希望在便签编辑器中输入快捷命令并回车触发：

```text
/t + 回车 -> 插入当前时间
/d + 回车 -> 插入当前日期
/i + 回车 -> 打开图片选择器
/f + 回车 -> 打开文件选择器
```

### 初始问题

物理键盘回车可以触发，但手机软键盘回车不会触发。

根因：

- 旧实现只靠 `EditText.setOnKeyListener(KEYCODE_ENTER)`。
- 大多数手机输入法不会发送硬件 `KeyEvent`。
- 软键盘通常通过 `InputConnection.commitText("\n")` 直接提交换行。

### 方案演进

最初考虑在 `TextWatcher.doAfterTextChanged` 中检测 `\n`。该方案被放弃。

放弃原因：

- `TextWatcher` 处于 IME 和 `EditText` 同步的敏感周期。
- 在回调中同步拆行、重建 View 树可能导致中文输入法组合态异常、光标乱跳、输入法状态错乱。
- 粘贴和输入难以在 `doAfterTextChanged` 中准确区分。

最终采用自定义 `NoteLineEditText`：

- 重写 `onCreateInputConnection()`。
- 使用 `InputConnectionWrapper`。
- 只拦截单字符 `commitText("\n")`。
- 捕获当时的 `selectionStart/selectionEnd`。
- `return true` 阻止换行进入 `Editable`。
- 使用 `post { onEnter(selectionStart, selectionEnd) }` 推迟到下一轮消息循环再拆行/执行命令。

同时保留：

- `setOnKeyListener(KEYCODE_ENTER)`：处理蓝牙键盘、平板键盘等真实物理按键。
- `sendKeyEvent(KEYCODE_ENTER)`：处理部分输入法模拟硬件回车。

### IME 细节

为了让输入法右下角显示“回车”而不是“完成/下一步”，`NoteLineEditText` 保持多行输入属性：

```kotlin
TYPE_CLASS_TEXT | TYPE_TEXT_FLAG_MULTI_LINE
IME_ACTION_NONE | IME_FLAG_NO_ENTER_ACTION
```

这样从输入法角度看它仍是多行输入框，但实际换行由编辑器控制。

## 六、多行粘贴拆成多个逻辑行

### 问题

用户提供了一个更新日志 txt：

```text
C:\Users\ASUS\Desktop\新建 文本文档.txt
```

检查后确认源文件本身正常：

- UTF-8。
- CRLF 换行。
- 共 50 行。

但复制到便签后，所有内容进入同一个 `EditText`，在软件里表现为一个超长行/单个块。

根因：编辑器是一行一个 `EditText`，但旧粘贴路径没有把多行文本转换成多个 `NoteParagraph`。

### 放弃方案

曾考虑在 `InputConnection.commitText()` 中拦截所有包含 `\n` 的文本并异步拆块。该方案被放弃。

原因：

- `commitText()` 是 IME 的同步事务。
- 如果返回 true 但 `Editable` 当前帧没有变化，输入法随后调用 `getTextBeforeCursor()` 时会读到旧状态，可能导致输入法缓冲区不同步。
- `InputConnection` 适合处理单个回车，不适合承担复杂粘贴处理。

### 最终方案

职责分离：

1. `InputConnectionWrapper` 只处理单字符软键盘回车。
2. 多行粘贴交给 `ViewCompat.setOnReceiveContentListener()`。
3. `onTextContextMenuItem(paste/pasteAsPlainText)` 做 ROM 兜底。

多行粘贴规则：

- 单行文本：交给系统默认粘贴。
- 多行文本：消费 payload，阻止整段文本进入单个 `EditText`。
- 捕获当前行 index 和光标 selection 快照。
- `post { insertMultilineText(...) }`。
- 统一换行：`\r\n`、`\r` 都转成 `\n`。
- 当前光标前内容 + 粘贴第一行留在当前 paragraph。
- 中间行插入为新的普通 paragraph。
- 粘贴最后一行 + 当前光标后内容组成最后一行。
- 最后只调用一次 `rebuildLines()`。

同时添加保护：

- 多行粘贴不会触发 `/t`、`/d`、`/i`、`/f` 快捷命令。
- 添加短时间去重，避免部分 ROM 同时触发 `OnReceiveContentListener` 和 `onTextContextMenuItem` 导致粘贴两次。
- 对少数输入法直接 `commitText("第一行\n第二行")` 的情况，先 `super.commitText()` 保证 IME 同步，再 `post` 规范化拆块。

## 七、退格与块合并

### 问题

软键盘退格不一定发送 `KEYCODE_DEL`，很多输入法会调用：

```kotlin
deleteSurroundingText(1, 0)
deleteSurroundingTextInCodePoints(1, 0)
```

如果只监听硬件按键，行首退格合并块在手机键盘上可能失效。

### 方案

在 `InputConnectionWrapper` 中补充拦截：

- `deleteSurroundingText(1, 0)`。
- `deleteSurroundingTextInCodePoints(1, 0)`。

当光标在行首时：

- `post { onBackspaceAtStart() }`。
- 消费事件。

这样软键盘和物理键盘都能触发行首块合并。

## 八、分割线与连续引用视觉

### 分割线

新增 `NoteParagraphType.DIVIDER`。

行为：

- 工具栏可以插入分割线。
- 分割线作为独立块，不保存正文文本。
- 导出 Markdown 时输出 `---`。
- 复制纯文本时也可表达为 `---`，但本轮最终 UI 去掉了“复制文本”按钮，保留底层能力。

### 连续引用

旧引用行每一行单独显示 `┃` 字符，连续引用时像多个分散标签。

本轮改成真正的左侧竖线样式：

- `QUOTE` 行左侧使用竖线 View。
- 相邻引用行视觉上更像一个整体引用块。
- 不改变数据结构，只改渲染。

## 九、附件打开与图片预览

### 9.1 文件点击崩溃

用户真机测试发现点击文件附件直接崩溃。

崩溃信息：

```text
java.lang.NullPointerException:
Attempt to invoke virtual method 'java.lang.String android.content.ContentResolver.getType(android.net.Uri)' on a null object reference
at android.content.ClipData.getMimeTypes
at android.content.ClipData.newUri
```

根因：

```kotlin
ClipData.newUri(null, "attachment", uri)
```

在 ColorOS 上会触发 NPE，因为 `ClipData.newUri()` 内部需要 `ContentResolver` 解析 MIME。

修复：

```kotlin
ClipData.newUri(context.contentResolver, "attachment", uri)
```

并增强文件打开 Intent：

- 使用 `FileProvider` 暴露私有附件。
- `ACTION_VIEW + setDataAndType(uri, mime)`。
- 增加 `ClipData`。
- 增加 `FLAG_GRANT_READ_URI_PERMISSION`。
- 查询目标应用并逐个 `grantUriPermission()`。
- 无匹配 MIME 时 fallback 到 `*/*`。
- 使用 `Intent.createChooser()` 打开。

### 9.2 点击与选中

初始改法让文件/图片单击直接打开，导致无法再选中附件块。

最终行为：

- 文件/图片：单击打开。
- 文件/图片：长按选中。
- 分割线：单击选中。

### 9.3 图片预览系统栏问题

最初使用 Compose `Dialog` 做图片预览，并尝试设置 Dialog window 的：

- `statusBarColor`。
- `navigationBarColor`。
- `WindowCompat.setDecorFitsSystemWindows(false)`。
- contrast enforcement。

但在 ColorOS 上底部小白条区域仍然显示灰/白，状态栏也不稳定。

参考 Techo 后确认：更稳的思路是让内容处在已有 Activity 的 edge-to-edge window 内，而不是新建 Dialog window。

最终方案：

- 取消图片预览 `Dialog`。
- 改成 `NoteEditorScreen` 根 `Box` 内部的全屏 overlay。
- overlay 背景直接 `Color.Black` 铺满。
- 图片内容使用 `systemBarsPadding()`，避免压到状态栏和手势条。
- 点击黑色背景或按返回关闭。

这样复用 `MainActivity.enableEdgeToEdge()` 和 Activity 已关闭导航栏 contrast 的行为，避开 Dialog window 在 ColorOS 上的系统栏灰底问题。

## 十、导出与 Markdown 互通

### 10.1 Markdown 导入导出

新增 `NoteMarkdownCodec`，用于 Markdown 与 `NoteDocument` 互转。

导入支持常见子集：

- `#` ~ `#####` -> H1 ~ H5。
- `>` -> 引用。
- fenced code -> 代码块。
- `- ` -> 无序列表。
- `1. ` -> 有序列表。
- `- [ ]` / `- [x]` -> 待办。
- `**粗体**`、`*斜体*`、`~~删除线~~`、`<u>下划线</u>` -> span。
- `---` / `***` -> 分割线。
- `![name](attachments/xxx)` -> 图片块。
- `[name](attachments/xxx)` -> 文件块。

导出规则：

- 无附件：导出 `.md`。
- 有附件：导出 ZIP，结构为：

```text
note.md
attachments/xxx
```

复杂 Markdown 暂不追求完整兼容，无法识别的内容按普通文本保留，避免再次把便签编辑器做成完整 Markdown 编辑器。

### 10.2 导出入口交互

曾短暂在更多菜单中加入：

- 导出。
- 导出 MD。
- 复制文本。

测试后认为入口太多，工具栏变重。

最终调整：

- 更多菜单只保留“导出”。
- 点击“导出”后，在工具栏上方显示轻量浮层。
- 浮层提供两个选择：
  - 默认文件。
  - Markdown。
- 浮层与工具栏同层，不使用全屏 scrim，不压暗页面和工具栏。
- 点击空白处或返回键关闭。

这样兼顾预测性返回手势和底部工具栏视觉层级。

## 十一、参考项目 Techo 的结论

本轮查看了参考项目 Techo。

Techo 的思路：

- 内容是块列表。
- 每个块是 `TechoItem`。
- 文本、待办、图片、分割线等都是不同块类型。
- 主页面用 RecyclerView 展示块。

但 Techo 没有直接解决我们的输入层问题：

- 它编辑文本块时使用单独的编辑面板。
- 没有 `InputConnectionWrapper`。
- 没有 `OnReceiveContentListener`。
- 没有多行粘贴拆成多个块的实现。

因此 Techo 可以证明“块级文档模型”的方向是对的，但输入法回车、多行粘贴、软键盘退格仍然需要我们自己处理。

## 十四、阶段结论

本轮最重要的结论是：便签编辑器的输入问题不能靠 `TextWatcher` 缝补，必须在输入源头分层处理。

最终架构：

```text
InputConnectionWrapper -> 只处理软键盘回车与退格
OnReceiveContentListener -> 处理现代 Android 粘贴/拖拽文本
onTextContextMenuItem -> 处理 ROM 粘贴兜底
TextWatcher -> 只同步普通文本，不做结构重建
NoteDocument -> 继续作为唯一结构化文档模型
```

这套方案比直接在文本变化后补救更复杂，但能避开 IME 组合态、光标同步、粘贴双触发和 ROM 差异带来的隐性问题，也更符合块级编辑器的长期方向。
