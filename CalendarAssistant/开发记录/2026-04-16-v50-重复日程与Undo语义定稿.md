# 2026-04-16 v50 重复日程与 Undo 语义定稿

## 一、文档目的与范围

本文件用于冻结“重复日程 + 内建动作 + Undo”语义，作为后续实现基线。

- 本文**不讨论**单日日程的普通增删改查（CRUD）。
- 本文聚焦：
  - 重复日程编辑/删除（this/future/all）
  - 内建动作（已完成/已用车/已取件/已取餐/已取票/已寄件/已检票）
  - Undo 一分钟回滚模型

---

## 二、已确认决议（冻结）

### 1) 重复日程 1~6 操作

以下操作统一对齐 Etar 级语义：

1. 编辑仅此一次
2. 编辑此及以后
3. 编辑全部
4. 删除仅此一次
5. 删除此及以后
6. 删除全部

### 2) 状态动作分类

- 立即过期动作（6 类）：
  - 已完成
  - 已用车
  - 已取件
  - 已取餐
  - 已取票
  - 已寄件

- 其中 `已用车/已取件/已取餐/已取票/已寄件` 仅用于单日日程。

- `已完成`：
  - 单日日程：直接更新结束时间为 `now`
  - 重复日程实例：按 Etar 的“仅此一次（exception）”语义处理

### 3) 已检票策略

- `已检票` 与 `已完成` 在业务上互斥（由规则/tag 分流）。
- `已检票`只走 APP 内规则状态流转：
  - 不改 `endTime`
  - 不写 Provider

### 4) Undo 策略

- 点击终态动作时，创建“新数据”并保存“备份数据”（一分钟有效）。
- 一分钟内触发 Undo：
  - 指针回退到备份数据
  - 删除新数据
- 超过一分钟自动销毁 Undo 记录。

## 三、项目现有实现 vs 拟改方案（总表）

### A. 重复日程编辑/删除

| 操作 | 项目现有实现 | 拟改方案（定稿） | 当前入口 |
|---|---|---|---|
| 编辑仅此一次 | 本地 `executeInstanceAction`：父排除实例 + 新增 exception + 同步 | 严格 Etar `MODIFY_SELECTED`：插入 exception（Provider 语义主导） | `AppRepository.modifySingleInstance` |
| 编辑此及以后 | 本地拆 past/future + Provider `splitFuture` | 严格 Etar `MODIFY_ALL_FOLLOWING`：截断旧系列 + 新建后半系列（COUNT/UNTIL 规则） | `AppRepository.modifyThisAndFutureInstances` |
| 编辑全部 | 本地 parent+child 全改，再 provider updateAll | 严格 Etar `MODIFY_ALL`（含 RRULE 移除时替换为单次） | `AppRepository.modifyAllInstances` |
| 删除仅此一次 | 空结果 action + Provider deleteThis/EXDATE 回退 | 严格 Etar `DELETE_SELECTED`：写取消 exception（`STATUS_CANCELED`） | `AppRepository.deleteSingleInstance` |
| 删除此及以后 | 暂无完整独立主流程 | 新增 `DELETE_ALL_FOLLOWING`（首实例=删整组；非首实例=截断） | 待新增 |
| 删除全部 | 本地删 parent/child/exception + Provider deleteAll | 保留能力，统一为 Etar 语义顺序与一致性 | `AppRepository.deleteEntireSeries` |

### B. 内建动作

| 操作 | 项目现有实现 | 拟改方案（定稿） | 说明 |
|---|---|---|---|
| 已完成（单日） | 改 `endTime=now` + 记录 originalEnd* | 保持 | 可 Undo |
| 已完成（重复实例） | 走 exception 管道 + 完成态 | 保持并收敛为 Etar this 语义 | 可 Undo（删 exception） |
| 已用车/已取件/已取餐/已取票/已寄件 | 通过规则动作落到 complete 路径 | 强约束为单日日程，直接 `endTime=now` | 不进入重复语义 |
| 已检票 | 当前会改本地 checkedIn，并可能带元数据联动 | 仅 APP 内规则状态，不改 endTime，不写 Provider | 与 completed 互斥 |

### C. Undo

| 场景 | 项目现有实现 | 拟改方案（定稿） |
|---|---|---|
| Undo 窗口 | 1 分钟（按 completedAt 判断） | 保持 1 分钟，但改为统一 token 机制 |
| 单日终态回滚 | 回填 originalEnd* | 保持语义，统一纳入 token |
| 重复实例回滚 | `restoreRecurringException` 删除 exception 并恢复父排除 | 保持并作为标准路径 |
| 已检票回滚 | 1 分钟内撤销检票 | 保持（APP 内状态回退，不触碰 Provider） |

## 四、Etar 对齐点（实现级要求）

### 1) MODIFY_SELECTED

- 通过插入 exception 实现单次编辑。
- 关键字段（Provider 层）应具备：
  - `ORIGINAL_SYNC_ID`
  - `ORIGINAL_INSTANCE_TIME`
  - `ORIGINAL_ALL_DAY`
  - `STATUS`

### 2) MODIFY_ALL_FOLLOWING

- 核心是“截断旧系列 + 创建新系列”。
- `updatePastEvents` 语义：
  - 若原规则是 `COUNT`：按实例展开数量精确拆分前后 COUNT
  - 否则：旧系列 `UNTIL = 当前实例开始前 1 秒`（UTC/all-day 规范化）

### 3) DELETE_SELECTED / DELETE_ALL_FOLLOWING / DELETE_ALL

- `DELETE_SELECTED`：插入 `STATUS_CANCELED` 的 exception
- `DELETE_ALL_FOLLOWING`：首实例删整组；非首实例截断旧系列
- `DELETE_ALL`：删除整组

## 五、Undo 统一模型（定稿）

为避免多入口分散判断，统一引入 Undo 记录（逻辑上是“指针 + 备份”）：

```kotlin
data class UndoToken(
    val tokenId: String,
    val targetKey: String,        // 单日=eventId；重复实例=seriesKey:instanceTime
    val actionType: String,       // complete/pickup/food/ticket/sender/taxi/checkin
    val beforeSnapshotId: String, // 备份数据标识
    val afterSnapshotId: String,  // 新数据标识
    val expireAtMillis: Long      // 创建后 + 60_000
)
```

### 执行规则

1. 触发动作：
   - 创建新数据
   - 记录 before/after 对应关系
   - 写入 token（60 秒）

2. 触发 Undo（未过期）：
   - 指针切回 before
   - 删除 after
   - 立刻销毁 token

3. token 过期：
   - 自动销毁
   - 不再允许回滚

## 六、边界场景定稿

### 1) Undo 后立刻再次点击终态动作

- 允许。
- 视为一次全新动作：
  - 旧 token 已失效
  - 新动作生成新 token（重新计时 60 秒）

### 2) 重复日程某一天实例的 Undo

- 对应 action 产生的 exception 为回滚锚点。
- Undo 时：
  - 删除该 exception（本地 + Provider）
  - 从父事件排除列表移除该 instanceKey
  - 刷新提醒与胶囊显示

### 3) 系列在回滚前已发生结构变化（被删/被重切分）

- Undo 返回失败并提示“实例已不可恢复”。
